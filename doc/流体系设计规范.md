# 流体系设计规范

> 定义 GitTributary 的 Flow 模块: 事件触发、任务编排、自动化执行与自然语言生成流的统一规范。
> 目标是保持足够小: 人能手写,AI 能生成,系统能验证,插件能扩展。

---

## 一、定位

Flow 是 GitTributary 的自动化层。

它不直接拥有 Git、复习、导出、脱敏、同步等业务逻辑,而是把这些能力编排起来:

```
Event 事件
  -> Flow 流匹配 trigger
  -> Guard 条件判断
  -> Step 动作执行
  -> Emit 输出新事件
  -> 其他 Flow 继续响应
```

一个 Flow 本身也是事件源。任何流开始、成功、失败、跳过、取消后都必须发出标准事件,供 UI、日志、审计和其他流订阅。

---

## 二、设计原则

| 原则 | 说明 |
| --- | --- |
| 小核心 | 核心只认 event / trigger / guard / step / policy / emit |
| 显式优先 | 危险动作、权限、并发、失败策略必须显式写出或使用默认值 |
| AI 友好 | 字段少、命名稳定、结构扁平,不用复杂脚本表达业务 |
| 可验证 | Flow 保存前必须 schema 校验、action 参数校验、权限校验 |
| 可观测 | 每次运行都有 run_id、输入事件、步骤结果、输出事件 |
| 可恢复 | 支持幂等键、重试、超时、失败处理,避免重复提交/重复推送 |
| 能扩展 | 新插件只需注册 event type 与 action type |

---

## 三、行业参考与取舍

FlowSpec 借鉴常见任务流表达方式,但只取最小公共集:

| 参考 | 借鉴点 | 舍弃点 |
| --- | --- | --- |
| GitHub Actions | `on` 触发、`steps` 顺序动作、`uses/with` 动作参数 | runner 矩阵、复杂表达式 |
| Temporal | 工作流运行记录、重试、超时、幂等、可恢复思路 | 代码即工作流的重模型 |
| BPMN | 事件、条件、分支、并行、人工确认 | 图形节点规范和复杂网关 |
| Node-RED / n8n | 事件进入节点、节点输出事件、动作插件化 | 任意连线图的高复杂度 |
| cron / systemd timer | 定时触发和 missed run 处理 | 仅限时间调度 |

结论: GitTributary 使用声明式 FlowSpec。复杂逻辑下沉到 action,Flow 只做编排。

## 四、核心概念

### Event

事件是系统中发生过的事实,不可变。

```json
{
  "id": "evt_01HY...",
  "type": "git.commit.created",
  "source": "gt-git",
  "time": "2026-06-24T22:30:00+08:00",
  "subject": "repo:/Users/mi/note",
  "data": {
    "repo": "/Users/mi/note",
    "branch": "main",
    "commit": "a1b2c3d"
  },
  "trace_id": "tr_01HY..."
}
```

字段规则:

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `id` | 是 | 全局唯一事件 ID |
| `type` | 是 | 点分命名,如 `git.commit.created` |
| `source` | 是 | 事件来源模块,如 `gt-git`、`gt-store`、`gt-flow` |
| `time` | 是 | RFC3339 时间 |
| `subject` | 否 | 事件主体,用于并发锁和审计 |
| `data` | 是 | JSON 对象 |
| `trace_id` | 否 | 串起一条响应链路 |

### Flow

流是一份持久化配置,描述如何响应事件。

一个 Flow 至少包含:

- `id`: 稳定 ID
- `name`: 用户可读名称
- `on`: 触发器
- `steps`: 执行动作

### Run

Run 是一次 Flow 执行实例。

同一个 Flow 可以有很多次 Run。Run 必须记录输入事件、步骤结果、状态和耗时。

状态机:

```
queued -> running -> succeeded
                  -> failed
                  -> skipped
                  -> canceled
```

## 五、FlowSpec 最小规范

推荐存储格式为 YAML。系统内部可等价转换为 JSON。

```yaml
version: 1
id: flow.auto_backup
name: 定时备份笔记
enabled: true

on:
  type: schedule
  cron: "*/30 * * * *"

when:
  all:
    - action: git.has_changes
      with:
        repo: "{{ workspace.active_repo }}"

policy:
  concurrency:
    key: "repo:{{ workspace.active_repo }}"
    mode: skip
  timeout: 10m
  retry:
    max: 2
    backoff: 30s
  on_error: stop

permissions:
  git: [status, commit, push]
  store: [read]
  network: true

steps:
  - id: commit
    use: git.commit_all
    with:
      repo: "{{ workspace.active_repo }}"
      message: "data: auto-sync {{ now | date:'YYYY-MM-DD HH:mm' }}"

  - id: push
    use: git.push
    with:
      repo: "{{ workspace.active_repo }}"
      remote: origin
      branch: "{{ steps.commit.branch }}"

emit:
  - type: flow.backup.completed
    data:
      repo: "{{ workspace.active_repo }}"
      commit: "{{ steps.commit.commit }}"
```

### 顶层字段

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `version` | 是 | 当前固定为 `1` |
| `id` | 是 | 点分稳定 ID,保存后不随名称变化 |
| `name` | 是 | 展示名称 |
| `description` | 否 | 简短说明 |
| `enabled` | 否 | 默认 `true` |
| `on` | 是 | 触发器 |
| `when` | 否 | 执行前条件 |
| `policy` | 否 | 并发、重试、超时、失败策略 |
| `permissions` | 否 | 权限声明 |
| `steps` | 是 | 动作列表 |
| `emit` | 否 | 成功后额外发出的业务事件 |
| `tags` | 否 | 分类标签 |

## 六、触发器 on

### 1. 事件触发

```yaml
on:
  type: event
  event: git.commit.created
  where:
    repo: "{{ workspace.active_repo }}"
    branch: main
```

用于 commit 后部署博客、store 变化后同步、复习完成后生成报告等。

### 2. 时间触发

```yaml
on:
  type: schedule
  cron: "0 9 * * *"
  timezone: Asia/Shanghai
  missed: skip
```

字段:

| 字段 | 说明 |
| --- | --- |
| `cron` | 五段 cron 表达式 |
| `timezone` | 默认使用系统时区 |
| `missed` | 应用休眠错过后如何处理:`skip` / `run_once` |

### 3. 文件监听触发

```yaml
on:
  type: file_watch
  path: "{{ workspace.active_repo }}/notes"
  events: [created, modified, deleted, renamed]
  glob: "**/*.md"
  debounce: 2s
```

### 4. 手动触发

```yaml
on:
  type: manual
  inputs:
    week:
      type: string
      required: false
```

用于用户点击按钮、命令面板、AI 临时执行。

### 5. Store 触发

```yaml
on:
  type: store_changed
  namespace: settings
  key: security.redaction.enabled
```

用于配置变化后自动刷新安全策略、同步配置等。

## 七、条件 when

条件只做布尔判断,不产生副作用。

```yaml
when:
  all:
    - expr: "{{ event.data.branch == 'main' }}"
    - action: git.has_changes
      with:
        repo: "{{ workspace.active_repo }}"
```

支持三种形式:

| 形式 | 说明 |
| --- | --- |
| `expr` | 简单表达式,只允许读取上下文,不允许执行命令 |
| `action` | 调用只读 predicate action,如 `git.has_changes` |
| `all` / `any` / `not` | 组合条件 |

表达式只允许访问:

- `event`
- `workspace`
- `settings`
- `profile`
- `steps.<id>`
- `now`

## 八、步骤 steps

步骤是顺序执行的动作。动作由模块注册。

```yaml
steps:
  - id: scan
    use: security.scan_secrets
    with:
      repo: "{{ workspace.active_repo }}"
      target: staged_diff

  - id: confirm
    use: ui.confirm
    if: "{{ steps.scan.risk != 'none' }}"
    with:
      title: 发现敏感信息
      message: "{{ steps.scan.summary }}"

  - id: push
    use: git.push
    with:
      repo: "{{ workspace.active_repo }}"
      remote: origin
      branch: main
```

步骤字段:

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `id` | 是 | 当前 Flow 内唯一,供后续步骤引用 |
| `use` | 是 | action type,如 `git.push` |
| `with` | 否 | action 参数 |
| `if` | 否 | 步骤级条件 |
| `timeout` | 否 | 覆盖全局超时 |
| `retry` | 否 | 覆盖全局重试 |
| `continue_on_error` | 否 | 失败后是否继续 |

### 顺序、并行、循环

默认顺序执行。复杂编排只保留三个扩展节点:

```yaml
steps:
  - id: export_all
    parallel:
      - id: export_pdf
        use: notes.export_pdf
      - id: export_index
        use: notes.export_index

  - id: redact_many
    foreach: "{{ steps.collect.files }}"
    as: file
    do:
      use: security.redact_file
      with:
        path: "{{ file }}"
```

规则:

- `parallel` 中的子步骤互不依赖。
- `foreach` 必须声明 `as`。
- 不支持任意跳转,避免 Flow 变成脚本语言。

## 九、策略 policy

```yaml
policy:
  concurrency:
    key: "repo:{{ workspace.active_repo }}"
    mode: skip
  timeout: 20m
  retry:
    max: 1
    backoff: 10s
  on_error: stop
```

| 字段 | 默认值 | 说明 |
| --- | --- | --- |
| `concurrency.key` | `flow:<id>` | 并发锁 |
| `concurrency.mode` | `skip` | `skip` / `queue` / `replace` |
| `timeout` | `30m` | 单次运行最大时长 |
| `retry.max` | `0` | 失败重试次数 |
| `retry.backoff` | `30s` | 重试间隔 |
| `on_error` | `stop` | `stop` / `continue` / `emit_failed` |

幂等建议:

- Git commit 类 action 应在无变更时返回 `skipped`,不创建空提交。
- Push / sync 类 action 必须能重复执行。
- 导出类 action 应使用确定性输出路径或 run_id 子目录。

## 十、权限 permissions

Flow 不能绕过权限。

```yaml
permissions:
  git: [status, log, commit, push]
  store: [read, write]
  files: [read, write]
  network: true
  secrets: false
  ui: [notify, confirm]
```

默认权限:

```yaml
permissions:
  git: [status, log]
  store: [read]
  files: [read]
  network: false
  secrets: false
  ui: [notify]
```

规则:

- 涉及 `push`、`delete`、`discard`、`secret`、`network` 的 Flow 首次启用必须提示用户确认。
- `secrets: true` 只允许读取脱敏视图,明文凭证只能由底层 action 内部使用。
- action 注册时必须声明自己需要的权限,Flow 保存时做静态校验。

## 十一、Action 注册规范

每个模块通过 action registry 暴露能力。

```json
{
  "type": "git.push",
  "label": "推送分支",
  "kind": "effect",
  "permissions": {
    "git": ["push"],
    "network": true
  },
  "input_schema": {
    "type": "object",
    "required": ["repo", "remote", "branch"],
    "properties": {
      "repo": { "type": "string" },
      "remote": { "type": "string" },
      "branch": { "type": "string" }
    }
  },
  "output_schema": {
    "type": "object",
    "properties": {
      "remote": { "type": "string" },
      "branch": { "type": "string" }
    }
  }
}
```

Action 分两类:

| 类型 | 说明 | 可用于 |
| --- | --- | --- |
| `predicate` | 只读判断,返回布尔或结构化判断结果 | `when` |
| `effect` | 产生副作用,如 commit、push、导出、脱敏 | `steps` |

首批建议内置 action:

| Action | 类型 | 说明 |
| --- | --- | --- |
| `git.has_changes` | predicate | 判断仓库是否有变更 |
| `git.commit_all` | effect | 暂存全部并提交 |
| `git.push` | effect | 推送分支 |
| `git.pull` | effect | 拉取分支 |
| `store.sync_now` | effect | 同步配置仓库 |
| `notes.review_due` | predicate | 判断是否有今日待复习 |
| `notes.export_review_pdf` | effect | 导出当周复习 PDF |
| `security.scan_secrets` | predicate | 扫描敏感信息 |
| `security.redact` | effect | 信息脱敏 |
| `ui.notify` | effect | 系统通知 |
| `ui.confirm` | effect | 用户确认 |

## 十二、事件命名规范

事件类型使用点分命名:

```
<domain>.<object>.<verb>
```

示例:

| 事件 | 说明 |
| --- | --- |
| `app.started` | 应用启动 |
| `git.repo.opened` | 仓库打开 |
| `git.status.changed` | 工作区状态变化 |
| `git.commit.created` | 新提交创建 |
| `git.push.completed` | 推送完成 |
| `store.key.changed` | KV 变更 |
| `file.created` | 文件创建 |
| `file.modified` | 文件修改 |
| `review.due.available` | 有待复习内容 |
| `flow.run.started` | 流开始运行 |
| `flow.run.succeeded` | 流运行成功 |
| `flow.run.failed` | 流运行失败 |

Flow 运行必须自动发出:

- `flow.run.queued`
- `flow.run.started`
- `flow.step.started`
- `flow.step.succeeded`
- `flow.step.failed`
- `flow.run.succeeded`
- `flow.run.failed`
- `flow.run.skipped`

## 十三、存储规范

Flow 属于用户配置,归数据中心管理。

推荐命名空间:

| 命名空间 | key | value |
| --- | --- | --- |
| `flows` | `flow.<id>` | FlowSpec |
| `flow-runs` | `run.<run_id>` | 最近运行摘要 |
| `flow-events` | `event.<event_id>` | 可选事件日志索引 |

保留策略:

- `flows`: public,可同步。
- `flow-runs`: local 或 cache,默认只保留最近 1000 条。
- `flow-events`: local 或 cache,大量事件不默认同步。
- 敏感输入不写入 FlowSpec,只引用 `private.credentials` 或底层 action 自行读取。

## 十四、模块边界

建议新增 Rust crate:

```
src-tauri/crates/gt-flow/
├── src/
│   ├── lib.rs        # FlowEngine 对外入口
│   ├── spec.rs       # FlowSpec / Trigger / Step / Policy 类型
│   ├── event.rs      # Event / EventBus / EventStore
│   ├── registry.rs   # ActionRegistry
│   ├── validate.rs   # schema + permission 校验
│   ├── runtime.rs    # 调度与执行
│   ├── schedule.rs   # cron / timer
│   └── template.rs   # 变量渲染
└── tests/
```

与现有模块关系:

```
gt-flow
  -> gt-store: 读取 FlowSpec、写运行摘要
  -> gt-git: 执行 Git action
  -> Tauri: emit UI 事件、请求确认
  -> 插件: 注册 action / event type
```

Tauri commands:

| command | 说明 |
| --- | --- |
| `flow_list` | 列出所有 Flow |
| `flow_get(id)` | 获取 FlowSpec |
| `flow_save(spec)` | 校验并保存 |
| `flow_enable(id, enabled)` | 启停 |
| `flow_run(id, inputs)` | 手动运行 |
| `flow_runs(id)` | 查看运行历史 |
| `flow_validate(spec)` | 仅校验 |
| `flow_actions()` | 获取可用 action registry |

## 十五、AI 自然语言生成规范

AI 生成 Flow 时必须遵守:

1. 只生成 FlowSpec,不生成任意脚本。
2. 优先使用已注册 action。
3. 不确定的字段使用占位变量,不要编造路径、token、远程名。
4. 涉及危险动作时自动补充 `permissions` 与 `ui.confirm`。
5. 生成后必须通过 `flow_validate`。
6. 如果自然语言缺少必要信息,返回待确认问题,而不是保存半成品。

AI 生成结果结构:

```json
{
  "status": "ready",
  "spec": {},
  "questions": []
}
```

或:

```json
{
  "status": "need_input",
  "spec": {},
  "questions": [
    {
      "field": "steps.1.with.remote",
      "question": "要推送到哪个远程? 常见值是 origin。"
    }
  ]
}
```

自然语言示例:

> 每天下午 6 点,如果笔记有变更,自动提交并推送。

AI 应生成:

```yaml
version: 1
id: flow.daily_evening_backup
name: 每日晚间备份
enabled: false
on:
  type: schedule
  cron: "0 18 * * *"
  timezone: Asia/Shanghai
when:
  all:
    - action: git.has_changes
      with:
        repo: "{{ workspace.active_repo }}"
policy:
  concurrency:
    key: "repo:{{ workspace.active_repo }}"
    mode: skip
  timeout: 10m
permissions:
  git: [status, commit, push]
  store: [read]
  network: true
steps:
  - id: commit
    use: git.commit_all
    with:
      repo: "{{ workspace.active_repo }}"
      message: "data: auto-sync {{ now | date:'YYYY-MM-DD HH:mm' }}"
  - id: push
    use: git.push
    with:
      repo: "{{ workspace.active_repo }}"
      remote: origin
      branch: "{{ steps.commit.branch }}"
```

注意这里 `enabled: false`: AI 新生成的高权限流默认不立即启用,由用户确认后开启。

## 十六、典型流示例

### 1. 时间触发: 今日复习提醒

```yaml
version: 1
id: flow.review_daily
name: 每日复习提醒
enabled: true
on:
  type: schedule
  cron: "0 9 * * *"
  timezone: Asia/Shanghai
when:
  all:
    - action: notes.review_due
      with:
        repo: "{{ workspace.active_repo }}"
steps:
  - id: notify
    use: ui.notify
    with:
      title: 今日待复习
      message: "有新的笔记需要复习。"
permissions:
  git: [log]
  store: [read]
  ui: [notify]
```

### 2. 时间触发: 导出当周复习 PDF

```yaml
version: 1
id: flow.export_weekly_review
name: 每周导出复习 PDF
enabled: true
on:
  type: schedule
  cron: "0 20 * * 0"
  timezone: Asia/Shanghai
policy:
  timeout: 20m
steps:
  - id: export
    use: notes.export_review_pdf
    with:
      repo: "{{ workspace.active_repo }}"
      range: this_week
      output: "{{ workspace.active_repo }}/exports/review-{{ now | date:'YYYY-WW' }}.pdf"
  - id: notify
    use: ui.notify
    with:
      title: 复习 PDF 已导出
      message: "{{ steps.export.output }}"
permissions:
  git: [log]
  store: [read]
  files: [read, write]
  ui: [notify]
```

### 3. Commit 后自动推送远程博客

```yaml
version: 1
id: flow.publish_blog_after_commit
name: 提交后发布博客
enabled: false
on:
  type: event
  event: git.commit.created
  where:
    repo: "{{ workspace.active_repo }}"
    branch: main
policy:
  concurrency:
    key: "publish:{{ workspace.active_repo }}"
    mode: queue
  timeout: 30m
permissions:
  git: [log, push]
  store: [read]
  network: true
  ui: [notify]
steps:
  - id: push_pages
    use: git.push
    with:
      repo: "{{ workspace.active_repo }}"
      remote: origin
      branch: gh-pages
  - id: notify
    use: ui.notify
    with:
      title: 博客已推送
      message: "gh-pages 已推送到 origin。"
emit:
  - type: publish.blog.completed
    data:
      repo: "{{ workspace.active_repo }}"
      branch: gh-pages
```

后续如果有静态站点构建 action,可在 `push_pages` 前插入 `site.build`。

### 4. 定时同步配置

```yaml
version: 1
id: flow.store_periodic_sync
name: 定时同步配置
enabled: true
on:
  type: schedule
  cron: "*/15 * * * *"
policy:
  concurrency:
    key: "store-sync"
    mode: skip
  timeout: 10m
  retry:
    max: 1
    backoff: 1m
permissions:
  store: [read, write]
  git: [status, commit, push]
  network: true
steps:
  - id: sync
    use: store.sync_now
```

### 5. 监听文件增删改并自动提交

```yaml
version: 1
id: flow.watch_notes_auto_commit
name: 监听笔记自动提交
enabled: false
on:
  type: file_watch
  path: "{{ workspace.active_repo }}"
  events: [created, modified, deleted, renamed]
  glob: "**/*.md"
  debounce: 5s
when:
  all:
    - action: git.has_changes
      with:
        repo: "{{ workspace.active_repo }}"
policy:
  concurrency:
    key: "repo:{{ workspace.active_repo }}"
    mode: queue
  timeout: 10m
permissions:
  git: [status, commit]
  store: [read]
  files: [read]
steps:
  - id: commit
    use: git.commit_all
    with:
      repo: "{{ workspace.active_repo }}"
      message: "notes: auto-save {{ now | date:'YYYY-MM-DD HH:mm' }}"
```

### 6. Push 前安全扫描与脱敏

```yaml
version: 1
id: flow.safe_push
name: 安全推送
enabled: true
on:
  type: manual
policy:
  concurrency:
    key: "repo:{{ workspace.active_repo }}"
    mode: skip
  timeout: 15m
permissions:
  git: [status, push]
  files: [read, write]
  store: [read]
  network: true
  ui: [confirm, notify]
steps:
  - id: scan
    use: security.scan_secrets
    with:
      repo: "{{ workspace.active_repo }}"
      target: staged_diff

  - id: redact
    use: security.redact
    if: "{{ steps.scan.risk == 'high' }}"
    with:
      repo: "{{ workspace.active_repo }}"
      findings: "{{ steps.scan.findings }}"

  - id: confirm
    use: ui.confirm
    if: "{{ steps.scan.risk != 'none' }}"
    with:
      title: 推送前确认
      message: "{{ steps.scan.summary }}"

  - id: push
    use: git.push
    with:
      repo: "{{ workspace.active_repo }}"
      remote: origin
      branch: "{{ event.inputs.branch || workspace.active_branch }}"
```

## 十七、最小实现路线

| 阶段 | 范围 | 目标 |
| --- | --- | --- |
| P0 | FlowSpec 类型、校验、store 持久化、手动运行 | 能保存并执行一个手动 Flow |
| P1 | ActionRegistry + git/store/ui 基础 action | 自动 commit/push/sync 可用 |
| P2 | 事件总线 + Flow run 事件 | Flow 能触发 Flow |
| P3 | schedule + file_watch | 时间触发、文件监听触发 |
| P4 | 运行历史、UI 面板、启停开关 | 用户可管理 |
| P5 | AI 生成 Flow + validate | 自然语言生成并确认启用 |
| P6 | 安全策略、权限提示、审计 | 高权限流可控 |

P0 最小闭环:

1. `FlowSpec` serde 类型。
2. `flow_validate(spec)`。
3. `flow_save(spec)` 写入 `flows` 命名空间。
4. `flow_run(id, inputs)` 顺序执行 `steps`。
5. 内置 `ui.notify` 与 `store.sync_now` 两个 action。

## 十八、一句话总结

> Flow 是 GitTributary 的神经系统: 事件进入,条件过滤,动作执行,再发出新事件。规范要小,动作要可注册,权限要显式,运行要可观测,这样它才能被人手写,也能被 AI 稳定生成。
