# 重构记录 · Tauri 命令层拆分

> 时间: 2026-07 (对应 src-tauri/src/lib.rs 拆分)
> 范围: `src-tauri/src/lib.rs`(Tauri 命令胶水层),不涉及 `crates/na-*` 业务 crate。
> 模式: Vibe Coding(对话式增量重构,每步搬代码后立即编译 + 测试验证)。

---

## 一、背景与目标

`src-tauri/src/lib.rs` 随着 Git / 数据中心 / 工作流 / 静态站点四大模块的命令不断累积,
膨胀到 **2589 行**(不含测试),所有 `#[tauri::command]`、认证解析、凭证绑定、Flow
执行器混在同一个文件里,难以定位和维护。

目标:

- 按命令所属领域拆成多个文件,不改变任何对外行为。
- 拆分前先补单元测试形成安全网,降低"改错了也不会有测试报错"的风险。
- 每个 `#[tauri::command]` 的函数名、参数、返回值签名保持不变,前端 `invoke()`
  调用零改动。
- 每步搬完立刻 `cargo build` + `cargo test` 验证,绿了才继续下一步。

## 二、重构前状态评估

拆分前先检查测试覆盖率,而不是直接动手改:

| 范围 | 单测数量 |
| --- | --- |
| `crates/na-git` | 18(`repo.rs` 等单测 + `tests/basic_test.rs`) |
| `crates/na-store` | 12(`tests/store_test.rs` + `tests/sync_test.rs`) |
| `crates/na-flow` | 10(`tests/flow_parse_test.rs`) |
| `crates/na-site` | 7(内置单测) |
| `src-tauri/src/lib.rs` | **0** |

四个业务 crate 测试覆盖良好,但胶水层(`lib.rs`)本身——认证优先级解析、路径安全
校验、错误分类、凭证绑定、Flow 执行器等 2589 行逻辑——**零测试覆盖**。直接对着
0 测试的文件做大重构风险很高,因此先补测试,再拆分。

## 三、安全网:补充单元测试

在 `lib.rs` 尾部新增 `#[cfg(test)] mod tests`,针对可独立验证的纯逻辑函数补测试:

- 认证优先级链(4 层 fallback:项目 token → 全局 token → SSH key → SSH agent)
- `resolve_auth_for_publish_target` 的凭据引用解析
- 错误分类 helper(`classify_*_error` 系列)
- 凭证展示摘要(`credential_summary_for_remote`)
- commit identity 解析优先级(远程覆盖 > 全局默认 > 应用兜底值)
- 路径安全校验(`is_empty_path`、`copy_dir_recursive`)
- Store key 命名 helper 的往返正确性

**关键技术决策**:测试函数需要构造 `AppState`,但 `tauri::State<T>` 的构造函数是
私有的(`pub struct State<'r, T>(&'r T)`,元组字段不可见),无法在没有 Tauri
runtime 的单测里直接构造。检查发现 `AppState` 内部字段是纯 Rust 类型
(`Mutex<Option<GitRepo>>` 等),`State<T>` 只是 `Deref<Target=T>` 包装、从未用到
Tauri 特有方法。于是把所有非 `#[tauri::command]` 的内部 helper 函数签名从
`State<'_, AppState>` 改为 `&AppState`——因为 `State: Deref<Target=AppState>`,
所有调用点 `&state` 依然编译通过(deref coercion),同时让这些函数可以在单测里
直接用普通 `&AppState` 调用。

补测试后运行 `cargo test --workspace`:**92 个测试全绿**(45 新增 + 47 原有),
形成安全网基线。

## 四、拆分方案与执行顺序

拆分原则:

1. 每个 `#[tauri::command]` 函数签名(名字/参数/返回值)完全不变。
2. 按"调用哪个业务 crate"分组,而不是按代码物理位置。
3. 认证解析、commit identity 解析等跨领域编排逻辑独立成模块,被多个命令组共用。
4. 单元测试跟着函数一起搬,不留在 `lib.rs`。
5. 每步拆完立刻 `cargo build --lib` + `cargo test --lib`,确认零警告、测试全绿才
   进入下一步。

执行顺序(共 11 次提取,每次都是独立可验证的 checkpoint):

| 序号 | 提取内容 | 落地位置 | 说明 |
| --- | --- | --- | --- |

| 2 | Workspace 命令 | `src/commands/workspace.rs` | 最小的独立命令组,验证拆分流程 |
| 3 | Store key 命名 helper | `src/keys.rs` | `project_token_key_for_path` 等 |
| 4 | 认证解析 | `src/auth.rs` | `resolve_auth` / `resolve_auth_for_publish_target` |

## 五、拆分中遇到的问题与处理

### 1. `tauri::State` 无法在单测中直接构造

见第三节。解决方式:把内部 helper 签名从 `State<'_, AppState>` 改为 `&AppState`,
依赖 deref coercion 保持调用点不变。

### 2. macOS TempDir 路径符号链接差异

`repo_path_for_repo` 的单测直接比较字符串路径,在 macOS 上因为 `/var` 是
`/private/var` 的符号链接导致断言失败(`TempDir` 返回的路径两种形式都可能出现)。
改为比较 `canonicalize()` 后的路径解决。

### 3. cargo 包缓存锁死

安全网测试首次运行时卡在 `Blocking waiting for file lock on package cache`——
这是项目已知问题(见 `doc/架构/开发调试与启动指南.md`),由此前反复启动/杀死
`cargo run` 进程留下的孤儿进程占用 `~/.cargo/.package-cache` 锁导致,和这次重构
的代码改动无关。清理僵死进程(`kill -9`)后恢复正常。

### 4. 跨模块函数迁移后的可见性调整

多个 helper 函数原来是 `lib.rs` 内的私有 `fn`,拆到子模块后需要改成 `pub(crate)`
才能被其它模块 `use crate::xxx` 引用,例如 `set_active_repo_state`、
`publish_flow_event`、`remote_url_for`。这类调整不改变函数行为,只影响可见性
修饰符。

## 八、遗留技术债(本次重构范围外)

以下问题在重构过程中被识别,但按照"只搬文件、不改行为"的原则未处理,留作后续
独立任务:

1. **认证优先级链重复三份**:`resolve_auth`(`auth.rs`)、
   `resolve_auth_for_publish_target`(`auth.rs`)、
   `credential_summary_for_remote`(`auth.rs`)几乎是同一套优先级判断的三次独立
   实现,只是返回类型不同。后续认证策略变化需要同时改三处。可考虑抽成一个
   返回统一"决策结果"的公共函数,各处按需转换返回类型。

2. **Flow 节点执行器是字符串匹配分发**:`AppFlowActionExecutor::execute`
   (`commands/flow.rs`)用 `match node.uses.as_str()` 做分发,这是
   `doc/工作流/流构造与执行技术实现.md` 中已记录的 P2 待办("真实 Action
   Registry")。每加一个新节点类型都要来改这个大 match。

3. **Git 写操作的事件发布不一致**:`commit_all`/`git_push`/`open_repo` 会发
   CloudEvent,但 `checkout_branch`/`delete_branch`/`create_branch` 不发,导致
   Flow 无法监听分支切换类事件。需要明确一条"哪些 Git 写操作应该发事件"的规则。

4. **`AppState` 用裸 `Mutex`,存在潜在锁顺序风险**:`repo`/`store`/`event_pool`/
   `node_registry` 四把独立锁,`commit_all` 等命令会依次拿 `repo` 锁又拿 `store`
   锁。目前单线程调用没有问题,但如果 Flow 未来支持并发执行多个节点,"先拿 A
   锁再拿 B 锁"的顺序如果在别处颠倒,存在死锁风险。

## 九、后续建议

- 新增 Tauri command 时,先判断属于哪个业务领域,放进对应的 `commands/*.rs`,
  不要再堆回 `lib.rs`。
- 跨领域编排逻辑(需要同时了解两个以上 crate 内部结构的逻辑)放在
  `src/` 根下的独立模块(参照 `auth.rs`/`identity.rs` 的模式),不要塞进某个
  具体的 `commands/*.rs`,避免其它命令模块产生意外依赖。
- 新增逻辑时同步补单元测试,保持"改了就有测试报错"的安全网,不要让新代码退回
  零覆盖状态。
- 第八节列出的技术债可分别拆成独立 spec 处理,不需要一次性解决。
