# GitTributary 插件规范

> 当前唯一来源是项目根目录 `plugins/`，其中的插件会随正式应用一起打包。

## 目录

```text
plugins/<plugin-name>/
├── manifest.json
├── frontend/
│   ├── package.json
│   ├── src/
│   └── dist/
└── backend/               # 可选 Rust 后端
    ├── Cargo.toml
    ├── src/
    └── target/release/
```

manifest 至少声明：插件 ID、版本、页面入口、权限，以及可选的 `rust-cdylib` 后端。
当前示例见 `plugins/repository-insights/manifest.json`。

## 开发

首次检出安装全部 workspace 依赖：

```bash
npm install
```

启动 Debug：

```bash
npm run tauri -- dev
```

默认 Debug 命令会自动构建 sidecar 和全部插件。也可以单独执行：

```bash
npm run plugin:build
npm run plugin:mvp:build
```

新增插件只需放入 `plugins/`。构建脚本会扫描 `plugins/*/manifest.json`，无需在根
`package.json` 登记插件名。插件前端必须属于 `plugins/*/frontend` npm workspace，
并提供 `build` 脚本。

## 构建与运行

```text
Debug   -> 从项目 plugins/ 发现插件
Release -> 从安装包 Resources/plugins/ 发现插件
安装后  -> 从 ~/.git-tributary/plugins/<id>/<version>/ 加载
```

正式构建会自动：

1. 构建全部插件和 sidecar。
2. 只收集 manifest、前端产物和当前平台动态库。
3. 将插件资源与 sidecar 放入应用安装包。

用户可以在插件页面安装、更新和卸载。卸载只删除用户安装目录，不修改安装包资源，
因此插件仍可再次安装。

## 平台能力

插件通过 Extension API 使用基础设施，目前开放：

- `repository:read`
- `git:read`
- `flow:read`

插件页面不能直接调用任意 Tauri command；前端 bridge 会把请求交给主应用校验。

## 当前边界

- 前端优先 React，后端优先 Rust。
- 只支持项目内插件，不支持远程下载或第三方发布。
- 不使用自定义插件包格式。
- 当前原生 Rust ABI 只服务 MVP，不视为稳定公开协议。
