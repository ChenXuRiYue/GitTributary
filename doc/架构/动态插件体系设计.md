# GitTributary 动态插件体系方向

> 当前先完成项目内插件 MVP，第三方生态以后再做。

## 当前架构

```text
plugins/ 源码
  -> 构建并随应用打包
  -> 用户在插件页安装
  -> React iframe 渲染插件页面
  -> Extension API 调用 Repository / Git / Flow
  -> gt-plugin-host sidecar 运行 Rust 后端
```

当前实现坚持以下边界：

- 主应用继续使用 React + Tauri + Rust。
- Repository、Git、Flow 是平台基础能力，由主应用统一提供。
- 插件前端独立构建，通过受控 bridge 使用平台能力。
- 插件 Rust 后端运行在 sidecar，不加载进 Tauri 主进程。
- 开发来源只有仓库根目录 `plugins/`，正式构建时作为内置资源进入安装包。
- 运行时支持安装、更新和卸载，实际加载目录是 `~/.git-tributary/plugins/`。

## 设计原则

1. 平台能力稳定，插件只依赖版本化 API。
2. 插件可以贡献页面，但不直接修改主应用 React 组件树。
3. 后端故障不应拖垮主进程。
4. 构建、安装和调试流程应由项目脚本统一处理。
5. 先用项目内插件验证契约，再讨论公开分发。

## 演进方向

1. 用更多项目内插件验证 manifest、生命周期和基础 API。
2. 补齐权限、日志、事件订阅和开发 SDK。
3. 完善多平台构建、签名和发布流程。
4. 有真实生态需求后，再增加远程索引、下载和开发者发布流程。

WASI、独立 Worker 或其他运行时都是后续候选方案，不在当前阶段提前锁定。

## 当前不做

- 不引入 `.gtplugin` 自定义包格式。
- 不建设独立插件市场服务。
- 不支持远程下载和第三方发布。
- 不承诺当前原生 Rust ABI 是长期公开协议。

当前可执行规范见 [插件规范](./插件规范.md)。
