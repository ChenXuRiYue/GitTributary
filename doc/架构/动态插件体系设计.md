# GitTributary 扩展架构

## 概念边界

| 概念 | 定义 | 典型内容 |
| --- | --- | --- |
| Core | 主应用内建、随应用直接运行的平台能力 | Git、Flow、数据、文件管理、插件管理 |
| 核心包 | Core 在 Rust 编译期依赖的能力包，不可运行时卸载 | `gt-git`、`gt-flow`、`gt-store`、`gt-files` |
| 插件 | 由 manifest 发现，可安装、更新、卸载的运行时扩展 | 发布等 `plugins/*` |

Git、Flow、数据和文件管理属于 Core，不作为插件。核心包是 Core 的内部实现单元，也不是插件。
`gt-files` 只负责受限路径下的目录列举、扫描、检索和文本读取；Markdown 候选规则、
站点渲染和 Pages 发布属于 `site-publisher` 插件。

侧边栏只渲染统一的 `WorkbenchItem`，不区分来源。Core 注册表和插件
`contributes.views` 都先转换成工作台项；来源类型只在注册与生命周期层使用。
插件也可通过 `contributes.flowNodes` 向 Core 的 Flow 节点注册表贡献动作。

## 当前链路

```text
Core（React + Tauri）
  ├── 编译期调用核心包
  └── 管理插件生命周期与 Extension API

plugins/ 源码
  -> 构建并随应用打包
  -> 安装到 ~/.git-tributary/plugins/
  -> iframe 页面 + Extension API
  -> gt-plugin-host sidecar + Rust cdylib
  -> FlowNode 注册表 + sidecar 节点执行
```

插件只依赖版本化 Extension API，不直接调用任意 Tauri command。插件后端在 sidecar
运行，故障不应拖垮主进程。

Flow 节点定义由 Core 统一登记。插件节点带插件所有者，不能覆盖 Core 或其他插件节点；
运行时由 `uses` 定位插件及 backend method。启动及插件生命周期变化时，宿主重新巡检
Core 和全部活跃插件，完整校验后原子替换节点池。节点目录直接读取注册中心归属，并关联
插件名称与版本展示，不从 `uses` 文本推断来源。

文件能力在编译期由 Core 调用 `gt-files`，在运行时通过 `files.list`、`files.scan`、
`files.search` 和 `files.readText` 提供给获得权限的插件。

## 当前范围

- 技术栈保持 React + Tauri + Rust。
- 插件唯一来源是仓库根目录 `plugins/`。
- 支持安装、更新和卸载，不使用 `.gtplugin` 格式。
- 暂不支持远程市场、第三方发布和稳定原生 Rust ABI。

后续先用项目内插件验证 API、生命周期和多平台构建，再决定公开分发方案。
具体开发约定见 [插件规范](./插件规范.md)。
