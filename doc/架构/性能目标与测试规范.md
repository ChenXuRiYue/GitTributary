# NoteAura 性能目标与测试规范

> 状态: 生效
> 基线日期: 2026-07-20
> 适用范围: Core、插件、Tauri IPC、Rust 核心包，以及会影响启动、导航、
> 仓库读取、文件扫描和列表渲染的基础设施。

---

## 一、目标与基本原则

NoteAura 面向可能包含大量文件、提交和附件的 Git 仓库。性能不是上线前的
一次性优化，而是每次修改都必须守住的功能契约。

本项目遵循以下原则:

1. 主线程优先。导航、输入、取消操作不能等待磁盘扫描、Git 遍历或序列化完成。
2. 按需加载。未显示的视图不得发起数据请求，首屏不得预取不可见的大数据集。
3. 有界工作。列表、IPC 结果、并发任务和缓存都必须有数量或体积上限。
4. 最新请求生效。仓库切换、视图切换和重复刷新后，过期响应不得覆盖新状态。
5. 可取消、可观测。超过 2 秒的工作必须可取消或转入后台，并记录分阶段耗时。
6. 修改即验证。每次代码修改至少通过快速性能门禁，合并前通过完整性能门禁。
7. 方案变更先通知。性能优化若改变架构、数据所有权或用户行为，必须先通知用户。

禁止用全应用遮罩等待后台任务。加载期间，侧边栏、导航、输入以及取消操作必须
保持可用。

## 二、统一性能预算

### 2.1 用户体验预算

除特别说明外，交互时延按 p95 判断。p95 采用 nearest-rank 算法，即排序后取
`ceil(样本数 * 0.95)` 对应的样本。

| 场景 | p95 预算 | 验收含义 |
| --- | ---: | --- |
| 输入或点击视觉反馈 | `<= 100 ms` | 按下后按钮、选中态或页面状态可见变化 |
| 页面壳或缓存快照出现 | `<= 100 ms` | 导航后立即看到目标布局，不等待完整数据 |
| 热缓存可用视图 | `<= 300 ms` | 首屏可读、主要操作可用 |
| 普通仓库冷加载 | `<= 1 s` | 活跃视图首屏数据可用 |
| 大型仓库冷加载 | `<= 2 s` | 活跃视图首屏数据可用 |
| 前端主线程 Long Task | `< 50 ms` | 单个任务不得阻塞一帧以上的连续交互 |

“可用视图”不是所有数据加载完成，而是首屏内容已经可读，首要操作已经可执行。
次要统计、详情和索引可以继续在后台完成。

### 2.2 请求与容量预算

| 项目 | 硬预算 |
| --- | ---: |
| 不可见视图发起的数据请求 | `0` |
| 同一仓库 generation 的完整 status 扫描 | `<= 1` |
| 过期响应覆盖当前状态 | `0` |
| 首批列表数据 | `100-200` 项，且 `<= 1 MiB` |
| 常驻 DOM 列表行 | `<= 300` |
| 磁盘密集型任务并发 | 默认 `1`，经测量后最多 `2` |

相同仓库、相同参数、相同 generation 的请求必须合并为 single-flight。队列必须
有界；达到上限后取消低优先级旧任务或拒绝新任务，不能无限积压。

### 2.3 基线回退预算

固定参考环境除满足绝对预算外，还必须满足:

- 同一 case 的 p95 不得比已批准基线回退超过 `20%`。
- 前端请求次数、status 扫描次数、过期覆盖次数不得回退。
- 结果条数或仓库规模变化时，必须同时报告吞吐量，不能只比较总时长。

若基线本身已超过绝对预算，测试必须失败并标记为待优化，不能通过更新基线把失败
合理化。只有需求或性能方案明确变化并完成通知、评审后，才允许调整预算或基线。

## 三、加载与任务分级

| 预计耗时 | UI 行为 | 任务要求 |
| --- | --- | --- |
| `< 300 ms` | 保留现有内容或轻量占位，避免 spinner 闪烁 | 可直接完成，但不得阻塞主线程 |
| `300 ms-1 s` | 在数据所属区域显示 skeleton 或 spinner | 异步执行，不影响其他页面操作 |
| `1-2 s` | 显示阶段或进度，如“读取状态/整理结果” | 分阶段埋点，失败可局部重试 |
| `> 2 s` | 显示后台任务状态和取消入口 | 可取消、可离开页面、完成后按需通知 |

加载状态必须归属具体区域，不能用全应用 loading 遮罩。快速请求的 loading 至少延迟
`300 ms` 再展示，避免短任务造成闪烁。离开视图后，未被其他消费者使用的任务必须
取消；仅取消前端等待不等于取消 Rust/libgit2 后台工作。

## 四、任务优先级与调度

| 优先级 | 工作类型 | 示例 |
| --- | --- | --- |
| P0 | 导航、输入、取消 | 切换 Core、选择仓库、停止扫描 |
| P1 | 当前视图首屏 | 当前仓库 status、历史第一页 |
| P2 | 用户明确选择的详情 | 文件 diff、提交详情、图片预览 |
| P3 | 非必要后台工作 | 数量统计、预取、索引、附件全量扫描 |

调度必须保证 P0/P1 不被 P3 占满。磁盘密集任务从全应用并发 `1` 开始，只有测试
证明吞吐提升且 P0/P1 延迟不回退时才可提高到 `2`。Git 和目录扫描主要受磁盘影响，
不得未经测量直接扩大线程池或引入无界并行。

## 五、测试数据集

性能 fixture 必须由脚本确定性生成，固定随机种子、目录形状、提交时间和文件内容。
测试报告记录 fixture 版本，不使用开发者日常仓库作为唯一依据。

| 等级 | 跟踪文件 | 工作区变化 | 提交历史 | 主要用途 |
| --- | ---: | ---: | ---: | --- |
| Small | 40 | 20 | 20 | 快速门禁和功能正确性 |
| Medium | 800 | 370 | 200 | 普通仓库冷/热加载预算 |
| Large | 4,000 | 1,600 | 800 | 大型仓库首屏、分页和容量上限 |
| Hostile | 300 | 3,050，48 层深目录 | 100 | 验证最坏目录形状和未跟踪遍历 |

Hostile 至少包含以下独立 case，避免多个极端条件混在一起后无法定位原因:

- 100 层深目录和大量未跟踪文件。
- 典型依赖/产物目录以及可忽略文件，验证 ignore 规则。
- 少量大文件，验证结果只传元数据或有界预览。
- 快速连续切换仓库和视图，验证取消及 latest-wins。
- 附件扫描与 Git 活跃视图同时运行，验证 P3 不阻塞 P0/P1。

Large fixture 生成成本较高，可以按 fixture 版本缓存；缓存只用于准备测试数据，不能
复用被测操作的应用缓存。冷加载 case 运行前清理应用运行时缓存，热缓存 case 则明确
执行一次预热。

## 六、测试分层

### 6.1 L0 静态契约门禁

每次修改都执行，目标是在秒级发现请求风暴和无界加载:

- Git 页面壳不得为装饰性统计请求 status、branches 或 log。
- 切换 Git 子视图不得重新 `open_repo`。
- 不可见视图不得挂载并发起数据请求。
- Changes 首次进入同一 generation 最多一次完整 status。
- History 首批必须有明确 limit，默认约 50 条，禁止请求全量历史。
- diff 只能在用户选中文件或提交后加载。
- 列表和 IPC 返回值必须有 limit 或体积保护。

### 6.2 L1 Rust 基准与集成测试

使用确定性 fixture 测量并校验:

- `status` 冷/热执行时间、文件数和序列化结果大小。
- `log` 第一页执行时间、分页边界和结果大小。
- 分支列表执行时间和结果数量。
- 深层未跟踪目录扫描时间及取消响应。
- 同 key 并发请求是否只执行一次底层扫描。
- 仓库切换后旧 generation 是否被丢弃。

时间测试至少预热 1 次、正式采样 20 次；CI 参考 runner 建议采样 30 次。原始样本、
p50、p95、最大值必须写入报告。开发机可以通过环境变量降低 fixture 规模，但降低
规模的结果只能用于诊断，不能作为合并验收结果。

### 6.3 L2 前端请求契约与渲染测试

- 记录进入 Git 页面、切换每个子视图时的 IPC 名称、参数、次数和顺序。
- 验证 StrictMode 下请求仍被去重，不以关闭 StrictMode 掩盖重复副作用。
- 验证首屏分批展示，常驻 DOM 行数不超过 300。
- 使用可控延迟制造乱序响应，验证 stale response 为 0。
- 使用 Long Tasks API 或等价采样，验证单个主线程任务 `< 50 ms`。
- 验证 300 ms 前不闪现 spinner，长任务只显示局部 loading 且可以取消。

### 6.4 L3 端到端性能场景

在构建产物或接近生产的 Tauri 环境执行:

1. 冷启动后进入 Git 变更视图。
2. 热缓存下在变更、历史、分支、远端间切换。
3. Large 仓库加载历史第一页并选择提交查看 diff。
4. 附件扫描进行中进入 Git 页面并持续导航、输入、取消。
5. 快速切换两个仓库，确认旧结果不回写且后台任务被取消。
6. 失败、超时和取消后重试，确认没有重复队列或全应用卡死。

端到端报告必须同时包含用户体验时延、IPC 请求数、Rust 操作时长、结果大小和
Long Task；只看总耗时无法区分排队、Git、序列化、IPC 与渲染问题。

### 6.5 L4 长稳与资源测试

发布前或涉及缓存、任务生命周期、文件扫描时执行:

- 连续切换视图、仓库和插件 30 分钟，检查内存、线程和句柄是否持续增长。
- 重复启动/取消附件扫描，确认队列能回落到 0。
- 缓存达到上限后的淘汰与重建。
- 应用切到后台及插件 iframe 销毁后，确认已无消费者的任务停止。

## 七、性能门禁

项目统一提供一个入口 `npm run perf:check`。门禁落地后不得用多个零散命令替代该
入口；内部可以编排静态契约、Rust 测试、前端测试和报告生成。

| 时机 | 必须执行 | 失败处理 |
| --- | --- | --- |
| 每次代码修改 | L0 + Small 快速 case | 不继续叠加修改，先定位或优化 |
| 修改 Git/扫描/IPC/缓存/并发/列表 | L0-L3 + Normal；按影响补 Large/Hostile | 任一硬预算失败不得提交 |
| 合并前 | 完整 L0-L3，固定参考环境 Normal + Large | 任一绝对或回退预算失败不得合并 |
| 发布前 | L0-L4 | 未达标不得发布 |

以下情况直接判定门禁失败:

- 任一硬预算超标。
- 测试超时、panic、样本缺失或报告字段不完整。
- 不可见请求、重复 status、stale overwrite 任一计数非零。
- 通过跳过测试、降低 fixture、放宽预算或更新基线规避失败。
- 性能相关修改没有提供变更前后对比。

测试受共享 CI runner 抖动影响时，可以自动复跑一次完整 case。第二次仍失败即视为
失败，不允许反复重跑挑选最佳结果。绝对时延门禁应绑定固定参考 runner；其他 runner
保留正确性、请求次数和相对趋势检查。

每份性能报告至少保存:提交 SHA、系统与硬件、构建模式、fixture 版本、预算版本、
原始样本摘要、p50/p95/max、相对基线、通过/失败原因。

## 八、指标采集规范

前端到 Rust 的一次操作使用同一个 `request_id` 串联。仓库路径不得明文写入日志，
使用稳定的匿名 `repo_id`。

| 字段 | 含义 |
| --- | --- |
| `request_id` | 单次用户或后台请求标识 |
| `repo_id` / `generation` | 匿名仓库标识及会话代次 |
| `operation` / `priority` | 操作名与 P0-P3 优先级 |
| `cache_hit` / `deduped` | 是否命中缓存、是否合并到已有任务 |
| `queue_ms` | 等待调度的时间 |
| `lock_ms` | 等待互斥资源的时间 |
| `git_ms` | libgit2/Git 或实际磁盘操作时间 |
| `serialize_ms` | Rust 结果序列化时间 |
| `ipc_ms` | IPC 传输和反序列化时间 |
| `render_ms` | 前端提交并呈现首屏的时间 |
| `total_ms` | 从请求发起到对应可用状态的总时间 |
| `result_items` / `result_bytes` | 结果条数与序列化体积 |
| `cancelled` / `stale` | 是否取消、是否因 generation 过期而丢弃 |
| `outcome` | success/error/timeout/cancelled |

生产日志默认只记录聚合和慢请求，不记录文件内容、diff、链接凭据和真实仓库路径。
诊断模式也必须遵守脱敏要求。

## 九、方案变更通知机制

自主性能优化可以在既定方案内进行，例如减少重复请求、补充分页、优化序列化或修复
无界列表。出现以下任一变化时，视为“方案变更”，必须在继续实现前及时通知用户:

- 改变 Git 会话、请求或缓存的数据所有权。
- 改变前后端接口、任务生命周期、并发/锁模型或取消语义。
- 新增状态管理、查询、虚拟列表、任务调度或持久化框架。
- 引入 SQLite、后台索引、常驻 worker 或新的缓存目录/格式。
- 改变用户可见的加载、刷新、一致性或错误恢复行为。
- 修改性能预算、fixture、采样方法或门禁豁免条件。

通知内容必须包含:

1. 当前方案及已测瓶颈。
2. 拟变更方案和变更原因。
3. 对用户体验、数据一致性、资源与维护成本的影响。
4. 回滚方式及新增/调整的测试 case。

通知不等于事后记录。除紧急止损外，应在实施方案变更前发出；紧急止损必须先停止
产生卡死或数据风险的行为，随后立即通知，不得借此扩大改动范围。

当前首阶段方案是:由 `GitPanel` 唯一持有仓库会话，子视图只请求当前可见内容；
后端重操作异步执行、读操作不长期持有全局锁；同 key 请求合并，generation 保证
latest-wins。首阶段不引入大型状态或测试框架。

当请求所有权、锁和取消问题解决后，才按实测需要评估:

- TanStack Query: 缓存、去重、失效、分页。
- TanStack Virtual: 列表超过约 200 行时的虚拟化。
- `tokio-util::CancellationToken`: Rust 长任务取消。
- SQLite: 跨仓库检索、分析或持久化索引。

框架不能替代正确的请求所有权和后端任务生命周期设计。磁盘密集扫描初期不引入
Rayon 式无界并行；Redux 也不用于解决后台阻塞问题。

## 十、完成定义

一项性能相关修改只有同时满足以下条件才算完成:

- 功能行为和错误处理正确。
- 对应层级的测试 case 已加入集合，而不只是手工验证一次。
- `npm run perf:check` 通过，并附变更前后报告。
- 绝对预算、相对回退预算和请求容量预算全部达标。
- loading 局部、异步、可取消，不影响 P0/P1 操作。
- 新增指标足以定位排队、锁、Git、序列化、IPC 和渲染阶段。
- 若方案发生变化，已按本规范通知用户并记录决策。
