# 代码质量与规模基线

## 一、目标

本基线用于控制 vibe coding 项目常见的无感膨胀：大文件继续增长、新增超大文件、临时豁免积累，以及一次变更跨越过多代码。它不限制正常功能增长，也不要求一次性清偿全部历史技术债。

核心策略是：

```text
记录当前债务
  -> 允许保持或改善
  -> 阻止新增和恶化
  -> 达到增长审查窗口后显式更新基线
```

版本化基线位于 `scripts/quality/quality-baseline.json`，普通检查不会自动修改它。检查报告写入 `quality-reports/latest/`，该目录不进入 Git。

## 二、当前硬门禁

`npm run quality` 执行规模基线和插件/Core 架构边界检查。基线包含以下规则：

1. 新代码文件不得超过 500 行。
2. 已进入基线的 500 行以上文件不得继续增长。
3. 相对基线累计增加超过 1500 行或 20 个代码文件时，必须重新审查项目规模。
4. 生产代码中的 `TODO`、`FIXME`、`HACK`、TypeScript/ESLint 抑制和 Clippy allow 不得增加。
5. 300 至 500 行文件进入观察列表，但暂不阻断合并。

这些数字是审查窗口，不是永久容量上限。达到窗口意味着需要确认增长落在哪些模块、是否伴随测试、是否应该拆包，而不是简单提高阈值。

## 三、Rust 编译级质量门禁

`npm run quality:rust` 会对三个 Rust workspace 依次执行：

```text
cargo fmt --check
cargo clippy --workspace --all-targets --all-features -- -D warnings
```

覆盖范围包括主 Tauri workspace、站点发布插件和附件管理插件。`site-publisher` 已将 `crates/gt-site` 纳入 workspace，因此其单元测试、格式与 Clippy 不再被上层插件清单遗漏。

首次克隆后，Tauri 的插件 sidecar 和资源尚未准备时，应先执行 `npm run plugin:mvp:build`，再运行完整 Rust 检查。

## 四、命令

```bash
# 日常快速质量检查，不编译 Rust
npm run quality

# Rust 格式和 Clippy
npm run quality:rust

# 全部质量检查
npm run quality:full

# 只检查当前规模基线
npm run quality:baseline

# 审查后显式更新基线
npm run quality:baseline:update
```

测试报告基础设施的自测 `npm run test:report:self` 同时覆盖基线比较逻辑。

## 五、基线更新规则

基线不得仅为了让 CI 变绿而更新。允许更新的情况：

1. 一个经过评审的功能确实需要扩大项目规模，并且新增职责和模块边界清晰。
2. 大文件完成拆分，需要把降低后的数字固化为更严格的新基线。
3. 一批历史抑制或 TODO 已清理，需要锁住改善结果。
4. 累计增长达到审查窗口，已经检查各 zone 的文件数、行数和测试投入。

更新流程：

1. 运行 `npm run quality:baseline` 阅读失败原因和 `quality-reports/latest/report.md`。
2. 在 PR 中说明增长模块、职责、测试和不拆分的理由。
3. 运行 `npm run quality:baseline:update`。
4. 审查 `quality-baseline.json` 的 zone 变化、超大文件列表和豁免标记变化。
5. 再次运行 `npm run quality:full` 和对应测试。

基线更新应作为清晰、可审查的提交内容，不能夹带无关格式化或大范围业务重写。

## 六、后续演进

当前阶段使用零外部依赖的规模与异味检测，先保证门禁稳定。后续按实际误报情况再引入：

- ESLint React Hooks 与类型感知规则。
- 前端模块依赖方向和循环依赖检测。
- 基于变更行的增量覆盖率。
- 函数级认知复杂度，而不仅是文件行数。
- 依赖漏洞、许可证和未使用依赖检查。

新增工具必须先以报告模式运行，确认信噪比后再升级为硬门禁。
