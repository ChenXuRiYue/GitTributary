# 数据层重构计划

> 状态：Data v2 的策略、领域入口、运行数据和插件容器已完成闭环；SQLite 查询投影按真实消费需求延后。
> 目标：业务通过领域 Repository 使用数据，数据层统一决定存储、同步、事件、安全和生命周期。

## 目标边界

NoteAura 不再把所有数据等同为 `(namespace, key, value)`。每个数据域必须先声明策略，再选择存储实现：

| 数据类别 | 事实来源 | 目标存储 | 默认同步 |
| --- | --- | --- | --- |
| 用户内容 | 用户仓库 | 文件系统 + Git | 是 |
| 可移植配置 | 配置仓库文档 | JSON/YAML 有界文件 | 是 |
| 本机状态 | 本机数据层 | 嵌入式 KV | 否 |
| 运行事件 | Run Journal | 分段 JSONL | 默认否 |
| 查询投影 | 可重建索引 | SQLite | 否 |
| 缓存 | 可重建结果 | KV/文件 | 否 |
| Secret | OS Keychain | 系统安全存储 | 永不同步 |
| 插件数据 | 插件作用域容器 | 按声明路由 | 默认本机 |

## 第一阶段：显式数据策略

第一阶段不迁移 JSONL 格式，也不修改现有 `null` 删除和时间戳合并语义。先建立唯一的数据策略入口：

```text
NamespacePolicyRegistry
  -> DataClass
  -> StorageClass
  -> SyncPolicy
  -> EventPolicy
  -> Sensitivity
  -> Rebuildable
```

当前内置策略：

| Namespace | 分类 | 同步 | 事件 |
| --- | --- | --- | --- |
| `settings` | PortableConfig | Required | Publish |
| `flows` | PortableConfig | Required | Publish |
| `sites` | PluginData | Optional | Publish |
| `workspace` | LocalState | LocalOnly | Private |
| `ui-state` | LocalState | LocalOnly | Private |
| `private.local` | LocalState | LocalOnly | Private |
| `private.credentials` / `secrets` | Secret | Never | Private |
| `plugin.*` | PluginData | LocalOnly | Private |
| 未登记 namespace | LocalState | LocalOnly | Private |

远程同步和 Store 变更事件必须查询同一份策略，禁止继续复制字符串判断。

## 后续阶段

### 第二阶段：领域 Repository（已完成统一入口）

已新增独立 `na-data` crate，作为应用统一数据入口：

```text
AppState
  -> DataHub                      应用级 Facade
     -> SettingsRepository        类型化设置
     -> WorkspaceRepository       本机工作区状态
     -> FlowDefinitionRepository  Flow 定义与目录
     -> CredentialsRepository     凭证领域接口（当前兼容旧后端）
     -> RemoteMetadataRepository  本机远程仓库元数据
     -> Profile/DynamicRepository 环境与受限动态数据
     -> Plugin Repositories       opaque payload 与宿主生命周期
     -> RunJournal/RunResultStore 运行事实与安全投影
     -> storage                    物理兼容后端（同 crate，不向业务暴露）
```

当前落地范围：

| 数据域 | 类型化能力 | 已迁移调用 |
| --- | --- | --- |
| Settings | `DataKey<T>`、`GIT_USERNAME`、`GIT_EMAIL` | Git 用户名和邮箱写入 |
| Workspace | `WorkspaceSnapshot`、初始化、同步、绑定/解绑仓库 | 启动初始化、仓库/分支状态、Workspace 查询 |
| Flow Definition | 定义与目录的读取、保存、删除 | Flow CRUD、启停、运行前查询和事件匹配 |
| Credentials | 全局/项目/配置仓库 Token、SSH Key、脱敏摘要 | 认证解析、凭证命令、远程同步 |
| Remote Metadata | 默认远程 URL、远程 Commit Identity | 远程配置聚合与身份解析 |

`AppState.data` 的类型是 `Mutex<DataHub>`。`DataHub` 已移除 `Deref`、`store()` 和 `store_mut()`，业务层不能退回裸 Store。通用动态命令只允许写 `LocalState` / `Cache`；Settings、Flow、Workspace、Secret 和插件数据必须经过对应 Repository。本阶段不修改已有用户数据格式，也不做双写。

当前 `CredentialsRepository` 是面向业务的稳定端口，底层暂时兼容旧 JSONL；第七阶段可在不影响调用方的情况下替换为 OS Keychain。同步 import/export 也通过 `DataHub` 基础设施端口执行，只处理策略明确允许同步的 namespace。

### 第三阶段：Flow 运行数据（已完成）

建立：

```text
RunStateStore
RunJournal
RunResultStore
```

运行事件先持久化，再发布给 Flow 和前端。日志按 `run_id` 和大小分段，封存后只读。

当前已落地第一批运行事实：

- 路径为本机 `runtime/flow-runs/v1/<encoded-run-id>/segment-NNNNNN.jsonl`，不进入 `na-data` namespace 和 Git Sync。
- Flow 执行副作用前追加 `run_started`，结束后追加 `run_completed`，终态落盘成功后才发布完成事件。
- 每个 run 使用单调 `seq`，单文件默认 4 MiB、单记录默认 1 MiB，超过阈值自动滚动新 segment。
- 每行带 CRC32；完整行损坏明确报错，仅在下一次追加前修复最后 segment 的崩溃残行。
- `run_id` 经过无损十六进制编码后才参与路径构造，不能形成路径穿越。
- Journal 只保存 ID、状态、时间等有界生命周期元数据，不保存 Flow inputs、event payload、节点 inputs/outputs 或错误明文。
- 提供 `reconcile_incomplete` 将只有 `run_started` 的遗留运行补记为 `run_abandoned/failed`；在取得 OS 单实例锁或确认 lease 失效前不得自动调用，避免第二个进程误伤仍在执行的 run。
- 默认保留最近 1000 个已结束运行；仍在 Running 的记录不会被自动删除。
- `flow_run_list` 用于发现运行历史，`flow_run_journal` 按 `run_id` 读取严格校验后的生命周期。
- 如果终态日志在副作用完成后写入失败，返回真实执行报告并发布 `flow.run.journal_failed`，避免客户端把它误判为执行失败后自动重试。

- `FlowRunObserver` 已把 Job/Node started/finished 元数据接入同一 Journal，并校验嵌套顺序。
- `RunResultStore` 保存不可覆盖的安全投影，仅含 run/flow/job/node ID、状态和时间，默认最大 1 MiB、保留 1000 条。
- 插件 stdout IPC frame 最大 1 MiB；单插件 Flow 节点结果进入运行报告前最大 256 KiB。
- Journal 或 Result 写入失败不改变真实执行报告，并发布不含业务 payload 的持久化失败事件。

当前写入串行性依赖应用进程内的 `flow_execution` 互斥锁。允许同时启动多个 NoteAura 实例前，必须先增加 OS 级单实例约束或按 run 文件锁；不能把当前实现视为多进程并发安全。

### 第四阶段：增量查询投影（按需启用）

SQLite 仅保存可重建查询索引。投影保存来源版本、事件游标、schema 版本和算法版本；正常路径增量更新，首次建立、损坏或版本升级时全量重建。

当前 `flow_run_list` 最多扫描 1000 个有界 run 目录，没有足以证明 SQLite 复杂度合理的查询消费者，因此不提前引入。出现跨大量 run 的组合筛选、稳定分页或聚合统计后再实现。

### 第五阶段：Portable Document 与 Git Sync v2

可移植配置改成按实体分片的 JSON/YAML 当前快照，Git 负责历史。同步基于 merge-base 做语义合并，删除使用显式 tombstone，不再依赖无限追加 JSONL 和文件顺序 LWW。

### 第六阶段：插件数据容器（已完成）

宿主管理插件数据容器、权限、配额、同步和生命周期；插件管理 opaque payload、schema、校验和迁移。插件默认只能访问 `plugin.<plugin-id>.*`，共享数据域必须显式授权和登记策略。

当前实现使用每插件一个物理 namespace，逻辑 subspace 编码进 key；支持 JSON `null`、默认 2048 keys / 256 KiB 单值 / 8 MiB 总 payload 配额和历史膨胀压缩。安装与升级保留 payload，卸载标记 orphan，重装重新 attach；`plugin-containers` 元数据仅宿主可见。热更新调用全链路携带 runtime generation，旧 iframe/session 会被拒绝。

### 第七阶段：Secret 与迁移收尾

凭证迁移到 OS Keychain。旧 JSONL 先备份、导入、校验 checksum，再原子切换；旧数据在观察期内只读保留，不长期双写。

## 强制规则

- 未登记数据默认 `LocalOnly + Private`。
- SQLite、缓存、同步游标和本机状态不得进入 Git。
- Git 同步文件必须是有界、可分片的文本文件。
- Secret 不得进入普通文件、日志、事件 payload 或同步仓库。
- `private.*`（`private.local` 除外）和 `secrets.*` 是不可降级的保留域，运行期策略注册不能把它们改成公开或可同步数据。
- 通用动态 Store API 必须拒绝 Secret namespace；Secret 只能通过受限的领域接口访问。
- 缓存和查询投影必须可删除、可校验、可重建。
- 数据写入成功后才能发布变更事件。
- 新业务必须先定义事实来源、生命周期、同步策略和 schema，再实现 Repository。

## 第一阶段验收

- `workspace`、`ui-state`、Secret 和未知插件 namespace 不会导出到配置仓库。
- 远端同名 local-only 文件不会导入本机 Store。
- `settings`、`flows`、`sites` 保持现有同步兼容。
- Store 事件发布不再使用独立字符串规则。
- 现有 Store API 和数据文件保持可读，现有测试继续通过。

## 第二阶段当前验收

- `DataHub` 成为应用状态中的统一数据入口。
- Settings、Workspace、Flow Definition、Credentials 与 Remote Metadata 的核心路径已通过类型化 Repository 访问。
- Repository 往返测试覆盖类型化设置、Workspace 快照、Flow 定义 CRUD、旧凭证格式兼容和远程元数据。
- 通用 Store Tauri API 拒绝并隐藏 Secret namespace，凭证明文不再能从该入口读取。
- 现有存储格式和同步行为保持兼容，但业务层已无裸 Store 入口。
- 插件数据、运行日志和安全结果均具有明确的 scope、大小、保留和失败语义。
