# 数据流与响应链路规范

> 定义 NoteAura 中「操作 → 状态持久化 → UI 响应」的标准链路。
> 目标:避免每个操作都手写重复的 store.set + refresh,用统一模式保证一致性与可维护性。

---

## 一、链路总览

```
用户操作(打开仓库/切换分支/修改设置/...)
    ↓
业务模块(na-git / 插件): 执行具体操作
    ↓
Tauri command 层: 操作成功后调用「聚合同步函数」
    ↓
数据中心(na-data): 持久化到 JSONL(一个调用同步所有相关 key)
    ↓ (后续) emit("store:changed", { namespace, key, value })
前端: useStoreKey() 声明式读取,自动响应变化
```

---

## 二、Rust 端规范

### 聚合同步函数(Sync Function)

每个「状态域」(workspace / settings / ...)有一个**聚合同步函数**,而非每个 command 手写多条 `store.set`:

```rust
// ✅ 正确:一个聚合调用
store.sync_workspace(Some(&path), Some(&branch));

// ❌ 错误:每个 command 里散写多条
store.set("workspace", "repo.active", json!(path));
store.set("workspace", "repo.branch", json!(branch));
store.set("workspace", "repo.recent", json!(recent));
```

**规则:**
- 新增持久化字段时 → 只改聚合函数内部,所有 caller 自动生效
- 聚合函数接受 `Option` 参数:传 `None` 表示该字段不变,传 `Some` 表示更新
- 聚合函数负责关联逻辑(如:设置 active_repo 时自动追加到 recent 列表)

### 已有的聚合同步函数

| 函数 | 作用域 | 触发时机 |
| --- | --- | --- |
| `store.sync_workspace(path, branch)` | workspace | open_repo / checkout_branch |
| `store.set_git_username/email/...` | credentials | 用户在设置面板修改 |
| 后续: `store.sync_ui_state(...)` | ui-state | 侧边栏宽度变化/视图切换 |

### 数据中心是唯一写入点

**所有持久化数据必须经过 na-data。** 业务模块不直接操作文件。

流向:
```
na-git(业务操作) → 返回结果 → Tauri command 层 → na-data(持久化)
                                                ↗
                               前端也只通过 invoke 调 na-data commands
```

### 数据引擎与 Git 模块边界

data-center 数据的存储、拉取、同步、合并必须只由数据引擎(`na-data`)实现。这里的 data-center 数据包括:

- `~/.noteaura/data/`
- `~/.noteaura/profiles/`
- `~/.noteaura/sync/`
- `~/.noteaura/credentials/`
- `~/.noteaura/cache/`

Git 模块(`na-git`)只负责 Git 仓库操作原语和 Git 事件信号:

- 仓库登记、远程配置、认证配置解析。
- `clone` / `open` / `fetch` / `pull` / `commit` / `push` 等 Git 操作接口。
- 仓库状态、远程状态、push/fetch 完成、认证失败等 Git 事件。

强制规则:

| 规则 | 说明 |
| --- | --- |
| 数据引擎拥有 data-center 生命周期 | data-center 数据的落盘、读取、快照、拉取、合并、同步状态更新都在 `na-data` 内完成 |
| Git 模块不读写 data-center | `na-git` 不直接读写 `.noteaura` 下的数据文件,也不决定 data-center 的存储结构 |
| 数据引擎可以调用 Git 原语 | `na-data` 同步远程配置时调用 `na-git` 的 clone/fetch/pull/commit/push 接口 |
| Git 事件只作为信号 | `na-git` 发出的事件只表示 Git 操作结果或仓库状态变化,不代表 data-center 已完成合并 |
| Tauri command 不绕过数据引擎 | 前端或 command 层不得直接调用 Git 原语去同步 data-center 文件 |
| 配置中心显式凭据 | data-center 配置中心仓库只能使用配置中心专用 Access Token,不得隐式依赖系统 Git 凭据、SSH Agent、普通 SSH Key 或 Git 全局 token |
| 配置中心环境维度 | 一个配置中心远程仓库可以承载多套独立环境配置,如 test/staging/prod;环境是数据空间维度,命名空间仍是一级配置域 |

标准流向:

```
na-data(sync 编排/数据合并)
    ↓ 调用
na-git(Git 操作原语: clone/fetch/pull/commit/push)
    ↓ 返回报告/发出 Git 事件
na-data(更新同步状态并 emit store:sync:* / store:changed)
```

## 三、前端规范

### useStoreKey Hook(声明式订阅)

```typescript
const [value, refresh] = useStoreKey<T>(namespace, key);
```

- 组件只声明「我需要哪个 key」,不关心数据从哪来、怎么同步
- 初始化时自动从 store 读取
- 后续(当 Tauri event 机制接入后)值变化时自动更新,组件零改动

### useWorkspace(快捷 hook)

```typescript
const { activeRepo, activeBranch, recentRepos, deviceId } = useWorkspace();
```

常用 workspace 状态的组合 hook,一次获取所有字段。

### 前端修改配置

前端修改配置也走 invoke:
```typescript
await invoke("store_set", { namespace: "settings", key: "theme", value: "dark" });
```

不允许前端直接读写本地文件。

## 四、数据分类与流向

| 数据类别 | 命名空间 | 可见性 | 写入时机 | 读取方式 |
| --- | --- | --- | --- | --- |
| 工作区状态(当前仓库/分支/最近) | workspace | public | 操作完成后自动 sync | useWorkspace() |
| 用户设置(主题/语言/策略) | settings | public | 设置面板修改时 | useStoreKey() |
| UI 布局(宽度/折叠/tab) | ui-state | public | 布局变化时 | useStoreKey() |
| 敏感凭证(token/key) | private.credentials | **private** | 设置面板修改时 | get_git_credentials |
| Git 默认远程选择 | private.local / git.default_remote_url | **private/local** | Git 设置面板修改时 | get_git_credentials |
| 配置中心专用 token | private.credentials / data_center.config_repo.token | **private** | 数据中心配置仓库时 | get_data_center_config_credential_status |
| 环境配置指针 | profiles/_active.json 或 sync/database.json active_environment_id | public/local 可配置 | 切换环境时 | store_active_environment |
| 插件配置 | plugins/<id> | public | 各插件自行 set | useStoreKey() |

---

`git.default_remote_url` 不能写入 public `settings` 或远端配置中心。它是本机 Git 模块的选择偏好,不能参与数据中心配置中心仓库的发现,否则会出现"先从远程同步默认远程,再用默认远程决定同步哪个远程"的循环依赖。

## 五、后续演进:Event-Driven Reactive

当前状态:前端主动 invoke 读取(pull 模式)。

后续目标:push 模式 — store.set 成功后 Tauri 自动 emit 事件,前端自动响应。

```rust
// na-data 内部:set 成功后 emit
fn set(&mut self, ns: &str, key: &str, value: Value) -> Result<()> {
    // ... 写入逻辑 ...
    // emit 事件(需要 AppHandle)
    app.emit("store:changed", StoreEvent { namespace: ns, key, value });
    Ok(())
}
```

```typescript
// 前端 useStoreKey 内部:监听事件
useEffect(() => {
  const unlisten = listen("store:changed", (event) => {
    if (event.payload.namespace === ns && event.payload.key === key) {
      setValue(event.payload.value);
    }
  });
  return () => { unlisten.then(fn => fn()); };
}, [ns, key]);
```

**这样实现后:**
- 后端任何模块调 `store.set` → 前端订阅该 key 的组件自动刷新
- 零轮询,零手动 refresh
- 组件代码无需修改(hook 内部升级即可)

## 六、设计原则总结

| 原则 | 说明 |
| --- | --- |
| 单一写入点 | 所有持久化经过 na-data,不直接碰文件 |
| 数据引擎同步 | data-center 数据的存储、拉取、同步和合并只由 na-data 实现 |
| Git 原语归 Git | Git 仓库操作原语和 Git 事件信号由 na-git 实现 |
| 聚合同步 | 一个函数同步一个域的所有相关 key,不散写 |
| 声明式读取 | 前端用 hook 声明需要的 key,不关心底层机制 |
| 分层解耦 | 业务模块产出状态 → store 负责持久化 → 前端负责展示 |
| 可演进 | 当前 pull 模式可平滑升级到 push(event-driven),无需改上层代码 |
| 不重复 | 新增字段只改聚合函数,不改 N 个 caller |
