# 事件池规范设计

> 事件池(Event Pool)是 GitTributary Flow 面向外界的统一事件信号入口。
> 它负责接收、标准化、记录和匹配事件,并把命中的 Flow 转换为待运行意图。
> 它是触发流的起点,但不是 Flow Runner,也不直接执行业务动作。

---

## 一、定位

事件池位于业务模块和 Flow 系统之间:

```text
Git / Store / File / App / UI / Scheduler / 后续外部集成
    ↓
事件池:接收事件、标准化 CloudEvent、记录、匹配 Flow
    ↓
Flow Run Intent
    ↓
Flow Runner:执行 workflow jobs / steps
```

一句话:

> 事件池 = Flow 的事件入口层 + 触发路由层。

事件池只回答三个问题:

1. 有哪些事件可以被 Flow 感知?
2. 某个事件来了以后,哪些 Flow 应该被触发?
3. 触发结果应该以什么运行意图交给后续 Runner?

---

## 二、职责边界

### 事件池负责

| 职责 | 说明 |
| --- | --- |
| 事件接收 | 为各模块提供统一 `publish`/`emit` 入口 |
| 事件标准化 | 将模块内信号转换为 CloudEvents 结构 |
| 事件登记 | 维护 GitTributary 当前支持的事件类型目录 |
| 事件记录 | 保存最近事件,用于调试、审计和 UI 观测 |
| 触发匹配 | 根据 enabled Flow 的 `on` 配置匹配事件 |
| 运行意图 | 生成 `FlowRunIntent`,交给 Runner 后续执行 |
| UI 通知 | 对前端广播事件进入、匹配、排队等状态 |

### 事件池不负责

| 非职责 | 归属 |
| --- | --- |
| 不执行 workflow step | Flow Runner |
| 不直接拥有 Git/Store/File 等业务逻辑 | 各业务模块 |
| 不直接读写任意业务文件 | 对应业务模块或数据中心 |
| 不做复杂表达式求值 | Flow 表达式引擎 |
| 不处理 action 权限执行 | Runner + Action Registry |
| 不保存完整长期运行历史 | `flow-runs` 按保留策略处理 |

## 三、核心概念

| 概念 | 说明 |
| --- | --- |
| `CloudEvent` | 事件池内部统一事件信封,遵循 CloudEvents 1.0 |
| `EventDefinition` | 一个可被 Flow 感知的事件类型定义 |
| `EventSource` | 事件来源,如 `gt-git`、`gt-store`、`scheduler` |
| `EventPublisher` | 事件发送者,通常是业务模块或 Tauri command 层 |
| `EventReceipt` | 事件被接收后的结果,包含事件 ID、匹配数量等 |
| `FlowTriggerMatch` | 一次事件命中某个 Flow 的匹配结果 |
| `FlowRunIntent` | 交给 Runner 的待运行意图,不是已执行结果 |

## 四、事件来源

事件池面向外界事件信号,所有模块都通过统一入口发送事件。

P0 建议先支持这些来源:

| 来源 | 典型事件 | 触发时机 |
| --- | --- | --- |
| App | `app.started` | 应用启动完成 |
| UI | `workflow_dispatch` | 用户手动点击运行 Flow |
| Git | `git.repo.opened` | 仓库打开成功 |
| Git | `git.commit.created` | commit 创建成功 |
| Git | `git.push.completed` | push 成功 |
| Store | `store.key.changed` | 数据中心 key 写入成功 |
| Flow | `flow.run.succeeded` / `flow.run.failed` | Flow 运行完成后 |

P1 再扩展:

| 来源 | 典型事件 | 说明 |
| --- | --- | --- |
| Scheduler | `schedule.tick` | 调度器产生 tick 后匹配 `on.schedule` |
| File Watcher | `file.created` / `file.modified` / `file.deleted` / `file.renamed` | 文件监听事件 |
| Sync | `store.sync.started` / `store.sync.completed` / `store.sync.failed` | 数据中心同步过程 |
| External | `external.webhook.received` | 后续外部事件入口 |

## 五、事件信封

事件池内部统一使用 CloudEvents。

```json
{
  "specversion": "1.0",
  "id": "evt_01HY...",
  "source": "gittributary://gt-git",
  "type": "git.commit.created",
  "time": "2026-06-27T20:30:00+08:00",
  "subject": "repo:/Users/mi/note",
  "datacontenttype": "application/json",
  "data": {
    "repo": "/Users/mi/note",
    "branch": "main",
    "commit": "a1b2c3d"
  }
}
```

字段规则:

| 字段 | 必填 | 规则 |
| --- | --- | --- |
| `specversion` | 是 | 固定 `1.0` |
| `id` | 是 | 全局唯一,建议 `evt_` 前缀 |
| `source` | 是 | 来源 URI,如 `gittributary://gt-git` |
| `type` | 是 | 点分事件名,如 `git.commit.created` |
| `time` | 建议 | RFC3339 时间 |
| `subject` | 建议 | 事件主体,用于审计、并发锁和 UI 展示 |
| `datacontenttype` | 建议 | 默认 `application/json` |
| `data` | 建议 | 业务数据,必须可序列化为 JSON |

事件命名规则:

```text
<domain>.<entity>.<verb>
```

示例:

- `git.repo.opened`
- `git.commit.created`
- `store.key.changed`
- `file.modified`
- `flow.run.succeeded`

## 六、事件登记

事件登记用于让 Flow 模块知道当前有哪些事件可用,也用于 UI 展示和 AI 生成 Flow 时选择触发器。

事件定义建议结构:

```json
{
  "type": "git.commit.created",
  "source": "gittributary://gt-git",
  "domain": "git",
  "description": "Git 仓库创建了新的提交",
  "stability": "stable",
  "data_schema": {
    "repo": "string",
    "branch": "string",
    "commit": "string"
  },
  "filters": ["repositories", "branches"],
  "example": {
    "repo": "/Users/mi/note",
    "branch": "main",
    "commit": "a1b2c3d"
  }
}
```

登记来源:

| 类型 | 说明 |
| --- | --- |
| 静态登记 | 各模块启动时注册自己能发出的事件 |
| Flow 动态订阅 | 保存/启用 Flow 后,从 `on` 字段提取它关心的事件 |
| 内置虚拟事件 | `workflow_dispatch`、`schedule`、`file_watch` 等由 Flow 层映射 |

P0 可以先将事件目录写死在 `gt-flow` 或 Tauri 状态层。
P1 再改为模块化 registry。

## 七、事件发送机制

所有模块发送事件时必须走统一入口,不直接调用 Runner。

Rust 侧建议形状:

```rust
event_pool.publish(EventDraft {
    source: "gittributary://gt-git",
    event_type: "git.commit.created",
    subject: Some(format!("repo:{repo_path}")),
    data: json!({
        "repo": repo_path,
        "branch": branch,
        "commit": commit_id,
    }),
});
```

Tauri command 调试入口建议:

```text
flow_emit_event(event)
flow_recent_events()
flow_event_catalog()
flow_match_event(event)
```

发送后的标准处理链路:

```text
publish(EventDraft)
    ↓
补齐 CloudEvent 字段(id/time/specversion/datacontenttype)
    ↓
校验 event.type 是否在事件目录中
    ↓
写入最近事件记录
    ↓
匹配 enabled Flow
    ↓
生成 FlowRunIntent
    ↓
广播 UI 事件
```

未知事件策略:

| 场景 | 处理 |
| --- | --- |
| P0 内部事件未登记 | 拒绝并返回错误,避免 typo 静默失败 |
| P1 外部事件未登记 | 可进入 `external.*` 灰度通道,但默认不触发高权限 Flow |

## 八、插件事件接入规范

事件池需要面向后续大量插件接入。插件事件必须被当作稳定 API 管理,不能让插件随意伪造内置事件、任意改变载荷结构或绕过权限。

### 8.1 插件事件定位

插件也是一种 `EventSource`,但它和内置模块隔离。

插件事件来源统一为:

```text
gittributary://plugin/<plugin_id>
```

插件事件类型默认使用插件命名空间:

```text
plugin.<plugin_id>.<entity>.<verb>
```

示例:

```text
plugin.review.card.due
plugin.ai.prompt.generated
plugin.export.pdf.completed
```

内置事件仍保留短域名:

```text
git.commit.created
store.key.changed
flow.run.succeeded
```

规则:

- 插件默认只能发布 `plugin.<plugin_id>.*` 事件。
- 插件不能发布 `git.*`、`store.*`、`flow.*` 等内置事件。
- 如确实需要提供领域级事件,必须由 GitTributary 内置适配层转发,不能由插件直接声明。

### 8.2 插件事件注册

插件必须先注册事件定义,再发布事件。事件定义可以来自插件 manifest,也可以由插件初始化时调用注册 API。

示例:

```json
{
  "type": "plugin.review.card.due",
  "source": "gittributary://plugin/review",
  "domain": "plugin.review",
  "description": "存在到期复习卡片",
  "stability": "stable",
  "version": "1.0.0",
  "data_schema_version": "1",
  "data_schema": {
    "repo": "string",
    "count": "number",
    "deck": "string"
  },
  "filters": ["repo", "deck"],
  "example": {
    "repo": "/Users/mi/note",
    "count": 12,
    "deck": "default"
  }
}
```

事件注册用于:

| 用途 | 说明 |
| --- | --- |
| 校验 | 未注册事件默认不能发布 |
| UI 展示 | 事件目录展示插件可触发事件 |
| AI 生成 | AI 根据事件定义生成 Flow 触发器 |
| Flow 诊断 | 插件卸载或事件废弃时能提示 Flow 失效原因 |

### 8.3 source 防伪

插件调用发布接口时不能自行指定最终 `source`。

建议接口形状:

```rust
event_pool.publish_plugin_event(
    plugin_id,
    PluginEventDraft {
        event_type: "plugin.review.card.due",
        subject: Some("repo:/Users/mi/note"),
        data: json!({
            "repo": "/Users/mi/note",
            "count": 12,
            "deck": "default",
        }),
    },
);
```

事件池负责补齐:

```text
source = gittributary://plugin/<plugin_id>
```

校验规则:

| 校验项 | 规则 |
| --- | --- |
| `event_type` 前缀 | 必须是 `plugin.<plugin_id>.` |
| `source` | 由事件池生成,插件不能覆盖 |
| `data` | 必须符合事件定义中的 schema |
| `subject` | 只能描述插件有权访问的资源 |

这样可以避免插件伪造内置事件,例如伪造 `git.commit.created` 来触发高权限 Flow。

### 8.4 插件事件权限

插件 manifest 需要声明事件能力。

示例:

```json
{
  "permissions": {
    "events.emit": ["plugin.review.*"],
    "events.subscribe": ["git.repo.opened", "store.key.changed"],
    "flow.trigger": true
  }
}
```

权限规则:

| 权限 | 说明 |
| --- | --- |
| `events.emit` | 插件允许发布的事件范围 |
| `events.subscribe` | 插件允许监听的事件范围 |
| `flow.trigger` | 插件事件是否允许触发 Flow |
| `flow.dispatch` | 插件是否允许主动发起 `workflow_dispatch` |

默认策略:

- 插件可以发布自己命名空间下的低风险事件。
- 插件不能监听所有事件。
- 插件不能默认触发包含高权限 action 的 Flow。
- 插件请求监听 `store.*`、`git.*`、`flow.*` 时需要用户确认。

### 8.5 版本与稳定性

插件事件一旦被 Flow 订阅,就成为面向用户配置的 API。

事件定义必须包含:

| 字段 | 说明 |
| --- | --- |
| `version` | 插件事件定义版本 |
| `stability` | `experimental` / `stable` / `deprecated` |
| `data_schema_version` | data 载荷结构版本 |
| `deprecated_by` | 可选,指向替代事件 |

兼容规则:

- `stable` 事件不能删除已有字段。
- `stable` 事件新增字段必须向后兼容。
- 语义变化较大时新增事件名或提升 schema version。
- `deprecated` 事件仍可发布一段时间,UI 需要提示迁移。

### 8.6 插件卸载与 Flow 失效

插件卸载后,其事件定义应从活动事件目录移除,但已有 Flow 不应被静默删除。

处理规则:

| 场景 | 处理 |
| --- | --- |
| Flow 订阅已卸载插件事件 | 标记为 `missing_event_definition` |
| 用户打开 Flow | UI 提示插件缺失 |
| 事件池匹配时 | 跳过并记录 skip reason |
| 插件重新安装 | 如果事件类型和 schema 兼容,自动恢复 |

### 8.7 限流与隔离

插件可能产生高频事件,事件池必须有隔离策略。

建议能力:

| 能力 | 说明 |
| --- | --- |
| per-plugin rate limit | 限制单插件每秒/每分钟事件数量 |
| debounce | 对文件、AI、同步类事件去抖 |
| TTL | 最近事件按时间和数量清理 |
| quota | 限制插件事件日志占用 |
| stats | 记录每个插件的事件数量、命中数量、失败数量 |
| disable switch | 用户可以禁用某插件事件触发 Flow |

## 九、Flow 触发匹配

Flow 的 `on` 字段声明它要订阅哪些事件。

示例:

```yaml
on:
  git.commit.created:
    branches: [main]
    repositories:
      - ${{ gt.workspace.active_repo }}
```

匹配过程:

1. 只匹配 `enabled = true` 的 Flow。
2. 事件 `type` 必须命中 `on.<event_type>`。
3. 如果 trigger 配置了过滤条件,继续检查 `data` 或运行时上下文。
4. 匹配成功后生成 `FlowRunIntent`。
5. 匹配失败时记录 `skip_reason`,便于 UI 调试。

P0 支持的过滤:

| 事件 | 过滤字段 | 数据来源 |
| --- | --- | --- |
| `git.repo.opened` | `repositories` | `event.data.repo` |
| `git.commit.created` | `repositories` / `branches` | `event.data.repo` / `event.data.branch` |
| `git.push.completed` | `repositories` / `branches` | `event.data.repo` / `event.data.branch` |
| `store.key.changed` | `namespace` / `keys` | `event.data.namespace` / `event.data.key` |
| `workflow_dispatch` | `inputs` | 用户输入 |

本地扩展触发器映射:

| Flow 写法 | 实际事件 |
| --- | --- |
| `workflow_dispatch` | UI 手动生成 `workflow_dispatch` |
| `schedule` | Scheduler 生成内部 tick 后匹配 cron |
| `file_watch` | File watcher 生成 `file.*` |
| `store_changed` | 映射为 `store.key.changed` |

## 十、FlowRunIntent

事件池不执行 Flow,只产生运行意图。

建议结构:

```json
{
  "id": "run_intent_01HY...",
  "flow_id": "flow.daily_evening_backup",
  "event_id": "evt_01HY...",
  "trigger": "git.commit.created",
  "reason": "event_matched",
  "created_at": "2026-06-27T20:30:00+08:00",
  "inputs": {},
  "event": {
    "type": "git.commit.created",
    "subject": "repo:/Users/mi/note"
  }
}
```

Runner 消费后再生成真正的 `flow.run.*` 事件:

- `flow.run.queued`
- `flow.run.started`
- `flow.run.succeeded`
- `flow.run.failed`
- `flow.run.skipped`

这些事件会重新进入事件池,因此可以支持“流触发流”。

防循环规则:

| 规则 | 说明 |
| --- | --- |
| 默认禁止同一个 `event_id` 重复触发同一个 `flow_id` | 避免重复运行 |
| `flow.run.*` 触发其他 Flow 时必须生成新的事件 ID | 保持事件链可追踪 |
| 同一事件链需要 `trace_id` 或 `correlation_id` | P1 引入,用于排查循环 |
| 连续触发深度需要上限 | P1 引入,默认例如 10 层 |

## 十一、持久化与保留

事件池数据归数据中心管理,但按数据性质分层。

| namespace | key | 说明 | 保留策略 |
| --- | --- | --- | --- |
| `flow-events` | `event.<event_id>` | 最近事件摘要 | 本地最近 1000 条 |
| `flow-runs` | `intent.<intent_id>` | 待运行意图或最近意图 | 本地最近 1000 条 |
| `flows` | `workflow.<flow_id>` | Flow 配置 | 可同步 |

原则:

- Flow 配置可以同步。
- 事件日志和运行历史默认只保留本地。
- 敏感数据不进入 `event.data`;只放引用或脱敏摘要。
- 事件池记录用于观测和调试,不是长期审计数据库。

## 十二、前端可视化

Flow 面板后续建议增加“事件池”视图,作为流中心起点。

建议区域:

| 区域 | 内容 |
| --- | --- |
| 事件目录 | 当前支持的事件类型、来源、可用过滤字段 |
| 最近事件 | 最近进入事件池的 CloudEvent 摘要 |
| 命中结果 | 某事件命中了哪些 Flow,哪些被跳过 |
| 手动投递 | 开发/调试用,手动发送测试事件 |
| Flow 订阅 | 当前选中 Flow 订阅了哪些事件 |
| 插件事件 | 插件发布的事件、权限状态、限流统计 |

一条最近事件建议展示:

```text
20:30:12  git.commit.created  repo:/Users/mi/note  matched: 2  skipped: 1
```

展开后展示:

- CloudEvent 摘要
- 匹配的 Flow
- 跳过原因
- 已生成的 Run Intent

## 十三、P0 实现路线

P0 目标:先让事件池成为可观测、可匹配、可触发的入口。

### P0.1 事件模型

- 定义 `CloudEvent` / `EventDraft` / `EventDefinition`
- 定义 `EventPool::publish`
- 生成事件 ID 和时间
- 保存最近事件到内存或 `flow-events`

### P0.2 事件目录

- 内置登记 `app.started`
- 内置登记 `workflow_dispatch`
- 内置登记 `git.repo.opened`
- 内置登记 `git.commit.created`
- 内置登记 `store.key.changed`

### P0.3 Flow 匹配

- 扫描 `flows` namespace 中 enabled Flow
- 用 `summary.triggers` 或后续解析结构匹配事件类型
- 支持 Git branch/repository 过滤
- 支持 Store namespace/key 过滤
- 输出 `FlowTriggerMatch`

### P0.4 运行意图

- 生成 `FlowRunIntent`
- 暂时不执行 step
- UI 可看到“该事件会触发哪些 Flow”

### P0.5 Tauri 命令

- `flow_event_catalog`
- `flow_recent_events`
- `flow_emit_event`
- `flow_match_event`

## 十四、P1 / P2 演进

### P1:Runner 接入

- Runner 消费 `FlowRunIntent`
- 执行本地 action registry
- 产生 `flow.run.*` 生命周期事件
- 支持 concurrency 和 skip/queue/replace
- 支持失败记录和 UI 通知

### P1:调度与监听

- `schedule` cron 转换为事件池输入
- `file_watch` 转换为 `file.*` 事件
- 支持 debounce、missed schedule 策略

### P2:外部事件

- 支持 webhook / deep link / plugin event
- 外部事件默认进入低权限域
- 高权限 Flow 需要用户显式确认
- 插件事件支持 manifest 注册、权限校验、限流和卸载诊断

## 十五、设计原则

| 原则 | 说明 |
| --- | --- |
| 入口统一 | 所有触发 Flow 的信号都进入事件池 |
| 信封统一 | 内部一律使用 CloudEvents |
| 业务外置 | 事件池不拥有 Git/Store/File 业务逻辑 |
| 先可观测 | 先能看到事件和匹配结果,再执行 Flow |
| 可解释 | 每次跳过和触发都要有 reason |
| 可回放 | 最近事件可以用于调试匹配,但默认不做长期审计 |
| 防循环 | Flow 产生的事件可以再触发 Flow,但必须有去重和深度限制 |
| 插件隔离 | 插件事件必须命名空间隔离、权限隔离、来源防伪 |

---

## 十六、最小闭环

事件池第一版完成后,应该能跑通这个闭环:

```text
用户打开仓库
    ↓
open_repo 成功
    ↓
发布 git.repo.opened CloudEvent
    ↓
事件池记录事件
    ↓
匹配 on.git.repo.opened 的 enabled Flow
    ↓
生成 FlowRunIntent
    ↓
前端事件池视图展示:事件、命中 Flow、待运行意图
```

这就是流中心真正的起点触发点。
