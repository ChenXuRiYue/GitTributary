# 流节点设计

> 本文讨论 GitTributary Flow 中“节点”的概念。
> 事件池负责把外部信号转换成 `FlowRunIntent`;流节点负责描述一次 Flow 运行中可执行、可观察、可失败、可恢复的动作单元。

---

## 一、定位

在 GitTributary 中:

```text
事件池(Event Pool)
    ↓ 产生 FlowRunIntent
Flow Runner
    ↓ 运行 Flow
Flow 节点(Node / Step)
    ↓ 调用本地 Action
业务模块(Git / Store / File / UI / Plugin)
```

节点不是 UI 图形里的“方块”那么简单,它首先是运行模型中的最小动作边界。

一句话:

> 流节点 = 有明确输入、输出、权限、失败边界和可观测状态的动作单元。

---

## 二、为什么需要节点

如果一个 Flow 只被看作一段整体脚本,失败时只能知道“发布失败”。这对本地自动化不够。

节点拆分后,系统能回答:

1. 哪一步失败?
2. 失败前已经完成了哪些动作?
3. 哪一步可以重试?
4. 哪一步需要高权限?
5. 哪一步产生了后续节点需要的输出?
6. UI 上应该如何展示运行进度?

因此节点拆分的原则不是“越细越好”,而是按这些边界拆:

| 边界 | 说明 |
| --- | --- |
| 可观测 | 用户需要知道它是否开始、成功、失败 |
| 可失败 | 它可能产生独立错误 |
| 可重试 | 它失败后可以单独重试或跳过 |
| 可授权 | 它需要明确权限,如 git push、文件写入、网络访问 |
| 有输入输出 | 它产出的结果会被后续节点引用 |
| 有业务语义 | 它对应一个用户能理解的动作 |

---

## 三、节点和 GitHub Actions 子集的关系

GitTributary Flow 采用 GitHub Actions workflow syntax 的本地子集。

因此第一阶段不另造一套节点 DSL,而是把 `jobs.<job_id>.steps[]` 视为节点。

示例:

```yaml
jobs:
  publish:
    runs-on: gittributary-local
    steps:
      - id: build
        uses: gittributary/notes/build-html@v1
      - id: push
        uses: gittributary/git/push@v1
```

对应:

| YAML 概念 | GitTributary 运行概念 |
| --- | --- |
| `workflow` | Flow 定义 |
| `on` | 触发入口 |
| `jobs.<job_id>` | 运行阶段 / 任务组 |
| `steps[]` | 节点 |
| `uses` | 本地 Action 引用 |
| `with` | 节点输入 |
| `outputs` | 节点输出 |

## 三点五、基于事件和节点构造 Flow 的初步逻辑

Flow 的构造不应该从空白 YAML 开始,而应该从两个已登记目录开始:

```text
事件目录(Event Catalog)
    ↓ 决定 on / trigger / filters
节点动作注册表(Node Registry)
    ↓ 决定 jobs.steps[] / uses / with / outputs
Flow 构造器(Flow Builder)
    ↓ 生成 Flow 草稿、节点预览和校验结果
Flow 保存
    ↓ 进入事件池匹配链路,等待产生 FlowRunIntent
```

一句话:

> 事件决定“什么时候运行”,节点决定“运行什么”,Flow 构造器负责把二者连接成可保存、可检查、可预览的 workflow。

### 3.5.1 输入

构造一个 Flow 至少需要这些输入:

| 输入 | 来源 | 作用 |
| --- | --- | --- |
| 目标描述 | 用户或 AI | 描述要自动化的业务目标 |
| 触发事件 | 事件池事件目录 | 写入 `on` 字段 |
| 触发过滤 | 事件定义 filters | 约束分支、路径、仓库、类型等 |
| 节点动作 | 节点注册表 | 写入 `jobs.<job_id>.steps[].uses` |
| 节点输入 | 事件 data、Store、Workspace、上游 outputs | 写入 `with` |
| 保存位置 | Flow 文件夹 | 用于 UI 组织和后续管理 |

P0 可以先从 UI 选择事件和节点动作;AI 生成 Flow 时也应使用同一套目录,不能凭空编造事件名和 `uses`。

### 3.5.2 构造步骤

P0 的 Flow 构造器可以按线性流程工作:

```text
选择触发事件
    ↓
读取事件定义 filters 和 data_schema
    ↓
选择或推荐节点动作链
    ↓
为每个节点生成 id、name、uses
    ↓
把事件 data、工作区变量、上游 outputs 绑定到 with
    ↓
生成 Flow YAML 草稿
    ↓
解析 YAML 为 FlowSummary
    ↓
编译 steps 为 FlowNodeSpec 列表
    ↓
展示事件入口、节点链和校验问题
    ↓
保存 Flow
```

这个流程里有两个“预览”:

| 预览 | 说明 |
| --- | --- |
| 事件预览 | 当前 Flow 会订阅哪个事件,过滤条件是什么 |
| 节点预览 | YAML 中的 steps 会被编译成哪些节点实例 |

只有两个预览都清晰,用户才应该启用 Flow。

### 3.5.3 节点链推荐规则

节点链不是任意排列。P0 可以先内置一组轻量顺序规则:

```text
trigger(虚拟)
context?
guard?
build?
validate?
sync?
git?
notify?
```

规则:

- `trigger` 来自事件池,不写入 YAML steps。
- `context` 通常放在最前面,负责解析路径、仓库、分支、输出目录。
- `guard` 用于提前结束无意义运行,例如没有变更时跳过。
- `build` 产生产物,例如 HTML、索引或导出文件。
- `validate` 在写入、提交、推送前校验产物。
- `sync` 把产物放到目标位置。
- `git` 执行提交、推送等仓库动作。
- `notify` 放在末尾,把结果反馈给用户。

构造器可以先按节点类型给出默认排序,再允许用户调整。P0 不做并行和 DAG,只做单 job 顺序 steps。

### 3.5.4 输入绑定规则

节点输入来自四类上下文:

| 上下文 | 示例 | 说明 |
| --- | --- | --- |
| 事件 | `${{ event.data.branch }}` | 来自触发事件的 CloudEvent data |
| 工作区 | `${{ gt.workspace.active_repo }}` | 当前打开仓库、工作区状态 |
| 数据中心 | `${{ gt.store.blog.repo }}` | 用户配置、目标仓库、偏好设置 |
| 上游节点 | `${{ steps.build.outputs.html_dir }}` | 前面节点产生的 outputs |

P0 实现可以先把这些表达式作为字符串保存,不立即求值。Runner 阶段再负责解析表达式和执行动作。

绑定约束:

- `with` 中引用的 `steps.<id>.outputs.*` 必须来自前置节点。
- 引用未登记输出时,保存前应给出警告或错误。
- 事件 data 字段应来自事件定义的 `data_schema`。
- Store 和 Workspace 引用可以先不强校验,但 UI 应标记为外部上下文。
- 敏感值不直接写入 YAML,只写引用。

### 3.5.5 校验规则

保存前至少校验:

| 校验 | P0 策略 |
| --- | --- |
| 事件存在 | `on` 中的事件必须能在事件目录中找到,虚拟事件除外 |
| 过滤字段存在 | 过滤字段应来自事件定义 filters |
| 节点动作存在 | 每个 `uses` 应能在节点注册表中找到 |
| step id 唯一 | 同一 job 下不能重复 |
| 必填输入 | 节点定义中的核心输入应能找到绑定值 |
| 输出引用顺序 | 只能引用前置节点输出 |
| 权限 | P0 先不考虑 |
| 执行器 | P0 先不执行,只构造和预览 |

这让 Flow 在保存阶段就能暴露大部分配置错误,而不是等到 Runner 执行时才失败。

### 3.5.6 构造结果

构造器最终产出三类结果:

```rust
pub struct FlowBuildDraft {
    pub raw_yaml: String,
    pub summary: FlowSummary,
    pub nodes: Vec<FlowNodeSpec>,
    pub diagnostics: Vec<FlowDiagnostic>,
}
```

| 字段 | 说明 |
| --- | --- |
| `raw_yaml` | 可保存的 workflow 文本 |
| `summary` | Flow 列表和详情页使用的摘要 |
| `nodes` | 节点管理页和后续 Runner 使用的节点实例 |
| `diagnostics` | 未注册事件、未知节点、缺少输入、非法引用等问题 |

P0 可以不落这个结构体,但前后端心智应按这个结果组织:先构造草稿,再校验预览,最后保存。

### 3.5.7 示例:由事件和节点构造发布 Flow

输入:

```text
事件: git.commit.created
过滤: branches = [main]
节点:
  - workspace/resolve-publish-context
  - notes/build-html
  - files/assert-exists
  - files/sync-dir
  - git/commit-all
  - git/push
  - ui/notify
```

构造结果:

```yaml
on:
  git.commit.created:
    branches: [main]

jobs:
  publish:
    runs-on: gittributary-local
    steps:
      - id: context
        uses: gittributary/workspace/resolve-publish-context@v1
      - id: build
        uses: gittributary/notes/build-html@v1
      - id: validate
        uses: gittributary/files/assert-exists@v1
      - id: sync
        uses: gittributary/files/sync-dir@v1
      - id: commit
        uses: gittributary/git/commit-all@v1
      - id: push
        uses: gittributary/git/push@v1
      - id: notify
        uses: gittributary/ui/notify@v1
```

保存后,事件池负责在 `git.commit.created` 到来时匹配这个 Flow,生成 `FlowRunIntent`;节点注册表负责把 steps 编译成可执行节点列表;Runner 后续再消费这些节点。

## 四、节点类型

P0 不需要在 YAML 中显式写 `type`,可以由 `uses` 或运行时语义推断。

| 类型 | 作用 | 示例 |
| --- | --- | --- |
| `trigger` | 事件池触发 Flow 的入口,不是 step 节点 | `git.commit.created` |
| `context` | 解析运行上下文,产出路径、分支、仓库等变量 | `workspace/resolve-publish-context` |
| `guard` | 判断是否继续运行,失败时通常 skip 或 fail | `git/has-changes` |
| `build` | 构建产物 | `notes/build-html` |
| `validate` | 校验产物或前置条件 | `files/assert-exists` |
| `sync` | 同步文件或数据 | `files/sync-dir` |
| `git` | 执行 Git 操作 | `git/commit-all`, `git/push` |
| `notify` | 通知 UI 或用户 | `ui/notify` |

后续如果做可视化编排,可以把这些类型映射成不同图标、颜色和默认表单。

## 五、节点数据模型

节点需要被解析成可执行结构,不能只停留在摘要。

建议结构:

```rust
pub struct FlowNodeSpec {
    pub id: String,
    pub name: Option<String>,
    pub job_id: String,
    pub uses: String,
    pub node_type: FlowNodeType,
    pub inputs: serde_json::Value,
    pub condition: Option<String>,
    pub timeout_minutes: Option<u64>,
    pub continue_on_error: bool,
    pub required_permissions: Vec<PermissionRequirement>,
}
```

运行时结构:

```rust
pub struct FlowNodeRun {
    pub run_id: String,
    pub flow_id: String,
    pub job_id: String,
    pub node_id: String,
    pub status: FlowNodeStatus,
    pub started_at: Option<String>,
    pub finished_at: Option<String>,
    pub inputs: serde_json::Value,
    pub outputs: serde_json::Value,
    pub error: Option<String>,
}
```

节点状态:

```text
pending
started
succeeded
failed
skipped
cancelled
```

## 六、示例:笔记系统构建 HTML 并推送博客仓库

用户目标:

> 笔记系统构建 HTML,并推送到远程博客仓库。

概念上这是 1 个 Flow:

```text
flow.publish_notes_blog
```

通常只需要 1 个 job:

```text
jobs.publish
```

推荐拆成 8 个节点:

```text
触发事件
    ↓
resolve-context
    ↓
build-html
    ↓
validate-html
    ↓
sync-to-blog-repo
    ↓
commit-blog-repo
    ↓
push-blog-repo
    ↓
notify-result
```

### 6.1 节点清单

| 顺序 | 节点 | 类型 | 作用 | 为什么独立 |
| --- | --- | --- | --- | --- |
| 0 | trigger | `trigger` | 由事件池触发 Flow | 记录这次运行来自哪个事件 |
| 1 | resolve-context | `context` | 解析笔记仓库、博客仓库、输出目录、目标分支 | 后续节点都依赖这些输出 |
| 2 | build-html | `build` | 把笔记内容构建成 HTML | 构建失败和 Git 失败应区分 |
| 3 | validate-html | `validate` | 检查 HTML 产物是否存在、是否为空 | 避免把坏产物推到远端 |
| 4 | sync-to-blog-repo | `sync` | 把 HTML 产物同步到博客仓库工作区 | 文件写入失败可单独定位 |
| 5 | commit-blog-repo | `git` | 在博客仓库提交变更 | 可能无变更、签名失败或提交失败 |
| 6 | push-blog-repo | `git` | 推送博客仓库到远程 | 需要网络和凭据,风险更高 |
| 7 | notify-result | `notify` | 通知用户发布结果 | 运行结束后的用户反馈 |

### 6.2 最小可执行版

如果先做 P0,可以压缩为 4 个节点:

```text
trigger
build-html
sync-output
commit-and-push
```

但这个拆法会损失可观测性:

- 不容易区分 commit 失败还是 push 失败。
- 不容易单独重试 push。
- 不容易在推送前校验产物。

因此推荐 P1 采用 6 到 8 个节点。

## 七、YAML 示例

```yaml
name: 发布笔记博客

gt:
  id: flow.publish_notes_blog
  enabled: false
  description: 构建笔记 HTML 并推送到博客仓库

on:
  workflow_dispatch:
  git.commit.created:
    repositories:
      - ${{ gt.workspace.active_repo }}
    branches: [main]

permissions:
  git: [status, commit, push]
  files: [read, write]
  store: [read]
  network: true
  ui: [notify]

jobs:
  publish:
    runs-on: gittributary-local
    steps:
      - id: context
        name: 解析发布上下文
        uses: gittributary/workspace/resolve-publish-context@v1
        with:
          source_repo: ${{ gt.workspace.active_repo }}
          target_repo: ${{ gt.store.blog.repo }}
          target_branch: main

      - id: build
        name: 构建 HTML
        uses: gittributary/notes/build-html@v1
        with:
          repo: ${{ steps.context.outputs.source_repo }}
          output: ${{ steps.context.outputs.output_dir }}

      - id: validate
        name: 校验 HTML 产物
        uses: gittributary/files/assert-exists@v1
        with:
          path: ${{ steps.build.outputs.html_dir }}
          non_empty: true

      - id: sync
        name: 同步到博客仓库
        uses: gittributary/files/sync-dir@v1
        with:
          from: ${{ steps.build.outputs.html_dir }}
          to: ${{ steps.context.outputs.target_repo }}/public

      - id: commit
        name: 提交博客产物
        uses: gittributary/git/commit-all@v1
        with:
          repo: ${{ steps.context.outputs.target_repo }}
          message: "publish: update notes blog"

      - id: push
        name: 推送博客仓库
        uses: gittributary/git/push@v1
        with:
          repo: ${{ steps.context.outputs.target_repo }}
          remote: origin
          branch: ${{ steps.context.outputs.target_branch }}

      - id: notify
        name: 通知发布完成
        uses: gittributary/ui/notify@v1
        with:
          title: 笔记博客已发布
          message: 已推送到远程博客仓库
```

## 八、节点输入输出

节点之间通过 `steps.<id>.outputs` 传递结果。

示例:

```text
context.outputs.source_repo
context.outputs.target_repo
context.outputs.target_branch
build.outputs.html_dir
commit.outputs.commit
push.outputs.remote
```

规则:

- 节点输出必须是 JSON 可序列化结构。
- 输出字段需要由 Action metadata 声明。
- 引用不存在的输出时,Flow 应失败,不能静默变成空字符串。
- 敏感输出不进入普通 `outputs`,只返回引用或脱敏摘要。

---

## 九、节点权限

节点权限来自两层:

1. Flow 顶层 `permissions` 声明允许范围。
2. Action metadata 声明自己需要的权限。

运行前校验:

```text
Flow permissions >= action required permissions
```

例如:

| 节点 | 需要权限 |
| --- | --- |
| `notes/build-html` | `files: [read, write]` |
| `files/sync-dir` | `files: [read, write]` |
| `git/commit-all` | `git: [status, commit]` |
| `git/push` | `git: [push]`, `network: true` |
| `ui/notify` | `ui: [notify]` |

高风险节点:

- `git/push`
- 删除文件类 action
- 使用 secrets 的 action
- 网络写入类 action

这些节点首次启用时应要求用户确认。

---

## 十、节点失败策略

默认策略:

```text
任一节点 failed -> 当前 job failed -> 当前 run failed
```

可选策略:

| 字段 | 说明 |
| --- | --- |
| `continue-on-error` | 当前节点失败后继续后续节点 |
| `if` | 条件不满足时节点 skipped |
| `timeout-minutes` | 超时后节点 failed |
| `retry` | 后续扩展,失败后重试 |

P0 可以先不实现 `retry`,但节点模型需要预留。

## 十一、Runner 事件

节点运行时需要产生事件,并重新进入事件池。

建议事件:

```text
flow.node.started
flow.node.succeeded
flow.node.failed
flow.node.skipped
```

事件 data:

```json
{
  "flow_id": "flow.publish_notes_blog",
  "run_id": "run_01HY...",
  "job_id": "publish",
  "node_id": "build",
  "uses": "gittributary/notes/build-html@v1"
}
```

这样 UI、日志、后续 Flow 都可以观察节点运行过程。

## 十二、UI 展示建议

Flow 详情页后续应展示:

| 区域 | 内容 |
| --- | --- |
| 节点拓扑 | 按顺序展示 trigger、context、build、validate、sync、git、notify |
| 节点详情 | 输入、输出、权限、Action metadata |
| 运行状态 | pending、started、succeeded、failed、skipped |
| 错误信息 | 失败节点的错误摘要和原始错误 |
| 重试入口 | 后续支持从失败节点重试 |

P0 可以先用线性列表。
P1 再做可视化节点图。

## 十三、开放问题

1. `trigger` 是否作为虚拟节点展示在每次运行中?
   - 建议展示,但不放进 YAML steps。
2. `commit` 和 `push` 是否应合并为 `publish-git`?
   - 建议不合并。提交和推送失败原因、权限和重试方式不同。
3. `build-html` 是否应属于内置 action?
   - 如果笔记系统是 GitTributary 核心能力,可以内置。
   - 如果未来支持多种笔记系统,更适合作为插件 action。
4. 节点是否支持并行?
   - P0 不支持。
   - P1 可通过 `jobs.needs` 做 job 级 DAG,step 仍保持顺序。

## 十四、阶段路线

### P0:线性节点运行

- 解析 `steps` 为 `FlowNodeSpec`
- 运行时生成 `FlowNodeRun`
- 支持顺序执行
- 支持 started/succeeded/failed/skipped 状态
- UI 展示线性节点列表

### P1:Action Registry

- Action metadata 声明 inputs / outputs / permissions
- 保存 Flow 时校验 action 是否存在
- Runner 根据 `uses` 调用本地 action

### P2:可视化节点图

- 按节点类型展示图标和状态
- 支持节点详情面板
- 支持失败节点重试
- 支持 AI 根据自然语言生成节点流

## 十五、结论

“笔记系统构建 HTML、推送远程博客仓库”应该是:
