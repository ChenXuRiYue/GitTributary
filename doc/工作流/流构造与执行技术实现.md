# 流构造与执行技术实现

> 本文记录 `gt-flow` 在“事件、节点管理、流构造”之后新增的构造草稿与顺序执行能力。
> 它面向后续维护者和 UI / AI 接入方,说明当前代码边界、核心数据结构、运行链路和未完成事项。

---

## 一、实现目标

本阶段目标是让工作流模块从“能解析和展示”推进到“能构造、能诊断、能执行”。

已落地能力:

| 能力 | 说明 |
| --- | --- |
| Flow 构造草稿 | 由事件、过滤条件、节点动作链生成 GitHub Actions 子集 YAML |
| 静态诊断 | 在保存或预览前发现未知事件、未知节点、缺输入、非法输出引用等问题 |
| 节点预览 | 将 `jobs.*.steps[]` 编译成 `FlowNodeSpec` |
| 顺序 Runner | 按 job / step 顺序执行本地 action |
| 表达式渲染 | 支持安全路径表达式,用于读取事件、输入、工作区和前置节点输出 |
| Tauri 命令 | 暴露 `flow_build_draft` 与 `flow_run` |
| 运行报告 | 返回 run / job / node 三级状态、输入、输出、错误和时间戳 |

当前仍保持的重要边界:

- 事件池只匹配 Flow 并生成 `FlowRunIntent`,不自动执行业务动作。
- `gt-flow` crate 不直接依赖 `gt-git` / `gt-store`;业务动作由宿主应用注入执行器。
- 默认不支持任意 shell `run`。
- 暂未实现并行 job、`if`、`needs`、`retry`、运行历史持久化。

## 三、Flow 构造器

### 3.1 输入结构

构造入口:

```rust
pub fn build_flow_draft(
    request: FlowBuildRequest,
    event_definitions: &[EventDefinition],
    registry: &FlowNodeRegistry,
) -> Result<FlowBuildDraft>
```

核心输入:

| 结构 | 说明 |
| --- | --- |
| `FlowBuildRequest` | Flow ID、名称、启停状态、触发器、job、steps |
| `FlowBuildTriggerRequest` | 触发事件类型与 filters |
| `FlowBuildStepRequest` | step id/name/uses/with |

构造器不会直接写 Store。它只生成草稿,再由现有 `flow_save` 保存。

### 3.2 输出结构

```rust
pub struct FlowBuildDraft {
    pub raw_yaml: String,
    pub summary: FlowSummary,
    pub nodes: Vec<FlowNodeSpec>,
    pub diagnostics: Vec<FlowDiagnostic>,
}
```

字段语义:

| 字段 | 说明 |
| --- | --- |
| `raw_yaml` | 可保存的 workflow YAML |
| `summary` | 由 `parse_workflow` 回读得到,用于列表和详情 |
| `nodes` | 由 `FlowNodeRegistry` 编译得到,用于节点预览和 Runner |
| `diagnostics` | 构造和保存前的静态诊断 |

### 3.3 构造流程

```text
FlowBuildRequest
  -> render_flow_yaml
  -> parse_workflow
  -> registry.compile_summary
  -> validate_flow_summary
  -> FlowBuildDraft
```

这样做的好处:

- 生成的 YAML 必须能被现有 parser 接受。
- UI / AI / 手写 YAML 最终走同一套摘要和节点编译逻辑。
- 构造器只负责草稿,不改变已保存 Flow。

### 3.4 诊断规则

当前诊断包含:

| code | severity | 说明 |
| --- | --- | --- |
| `unknown_event` | error | `on` 中事件未登记 |
| `unknown_event_filter` | warning | filter 不在事件定义 filters 中 |
| `unknown_node` | error | step `uses` 未登记 |
| `missing_input` | warning | 节点声明了输入 schema,但 step 未提供 |
| `duplicate_step_id` | error | 同一 job 下 step id 重复 |
| `invalid_step_reference_order` | error | 引用了后置或不存在的 step 输出 |
| `unknown_step_output` | error | 引用了已知 step 未声明的 output |

虚拟触发器处理:

- `schedule`、`file_watch` 是 Flow 层虚拟触发器,不要求事件目录中存在同名 CloudEvent。
- `store_changed` 会映射到 `store.key.changed` 做事件定义校验。

## 五、Runner

### 5.1 入口

```rust
pub fn run_flow_with_executor(
    record: &FlowRecord,
    request: FlowRunRequest,
    registry: &FlowNodeRegistry,
    workspace: Value,
    executor: &mut impl FlowActionExecutor,
) -> FlowRunReport
```

`gt-flow` 只负责编排。真正动作由 trait 注入:

```rust
pub trait FlowActionExecutor {
    fn execute(
        &mut self,
        node: &FlowNodeSpec,
        inputs: &BTreeMap<String, String>,
        context: &FlowExecutionContext,
    ) -> Result<FlowActionOutcome>;
}
```

### 5.2 运行结构

Runner 返回:

```rust
pub struct FlowRunReport {
    pub run_id: String,
    pub flow_id: String,
    pub flow_name: String,
    pub status: FlowRunStatus,
    pub trigger: String,
    pub reason: String,
    pub started_at: String,
    pub finished_at: String,
    pub jobs: Vec<FlowJobRun>,
    pub event: Value,
    pub inputs: Value,
    pub error: Option<String>,
}
```

状态枚举:

```text
pending
running
succeeded
failed
skipped
```

当前 P0 执行策略:

```text
FlowRecord
  -> enabled 检查
  -> compile_record
  -> 按 jobs 顺序执行
  -> 按 steps 顺序执行
  -> 任一节点 failed 后停止当前 run
  -> 返回 FlowRunReport
```

如果 Flow 被禁用:

- Runner 返回 `FlowRunStatus::Skipped`
- Tauri 命令发布 `flow.run.skipped`

### 5.3 表达式支持

Runner 支持字符串内表达式替换:

```yaml
repo: ${{ gt.workspace.active_repo }}
branch: ${{ event.data.branch }}
output: ${{ steps.context.outputs.output_dir }}
message: "data: auto-sync ${{ gt.now }}"
```

当前支持上下文:

| 上下文 | 示例 |
| --- | --- |
| `gt.now` | `${{ gt.now }}` |
| `gt.workspace.active_repo` | `${{ gt.workspace.active_repo }}` |
| `gt.workspace.active_branch` | `${{ gt.workspace.active_branch }}` |
| `event.*` | `${{ event.data.branch }}` |
| `inputs.*` | `${{ inputs.branch }}` |
| `steps.*.outputs.*` | `${{ steps.build.outputs.html_dir }}` |

不支持:

- 函数调用,例如 `gt.git.has_changes(repo)`
- 逻辑表达式,例如 `inputs.branch || main`
- 条件判断,例如 `if`
- shell 执行
- 任意文件读取

表达式解析失败时,当前节点 failed,不会静默渲染为空字符串。

## 六、Tauri 命令

### 6.1 `flow_build_draft`

用途:

- UI 选择事件和节点后生成 YAML 草稿。
- AI 生成 Flow 后做统一预览和诊断。

签名:

```rust
fn flow_build_draft(
    request: FlowBuildRequest,
    state: State<'_, AppState>,
) -> Result<FlowBuildDraft, String>
```

内部依赖:

- `state.event_pool.catalog()`
- `state.node_registry`
- `gt_flow::build_flow_draft`

### 6.2 `flow_run`

用途:

- 手动执行已保存 Flow。
- 后续消费 `FlowRunIntent`。

签名:

```rust
fn flow_run(
    id: String,
    request: Option<FlowRunRequest>,
    state: State<'_, AppState>,
) -> Result<FlowRunReport, String>
```

流程:

```text
读取 flows/workflow.<id>
  -> 构造 workspace context
  -> AppFlowActionExecutor
  -> gt_flow::run_flow_with_executor
  -> 发布 flow.run.succeeded / failed / skipped
  -> 返回 FlowRunReport
```

注意:

- `flow_emit_event` 仍只发布事件并返回 `run_intents`,不会自动调用 `flow_run`。
- 自动消费事件意图应放到后续队列 / 调度器中实现。

## 七、应用级 Action 执行器

`AppFlowActionExecutor` 位于 `src-tauri/src/lib.rs`。

它是 `gt-flow` 和业务 crate 的连接层:

| action | 实现方式 |
| --- | --- |
| workspace context | 读取 `FlowExecutionContext.workspace` 和 step inputs |
| build-html | 返回 `html_dir = output`,等待真实构建系统接入 |
| assert-exists | `std::fs` 检查路径存在和可选非空 |
| sync-dir | 简单递归复制目录 |
| git commit-all | `GitRepo::open`、`stage_all`、`commit` |
| git push | `GitRepo::open`、`push`、复用 `resolve_auth` |
| store sync-now | 复用数据中心同步链路 |
| ui notify | 写入节点 message,暂不弹系统通知 |

`git/commit-all` 对无变更的处理:

- 捕获 `GitError::NothingToCommit`
- 节点标记为 `skipped`
- 输出 `branch`, `commit = null`

## 八、运行事件

事件目录新增:

| 事件 | 说明 |
| --- | --- |
| `flow.run.succeeded` | Flow 执行成功 |
| `flow.run.failed` | Flow 执行失败 |
| `flow.run.skipped` | Flow 被跳过 |

Runner 完成后由 Tauri 命令发布这些事件。

事件 data:

```json
{
  "flow_id": "flow.publish_notes_blog",
  "run_id": "run_...",
  "status": "succeeded"
}
```

事件池收到这些事件后仍然只匹配并生成新的 `FlowRunIntent`。是否自动执行下游 Flow,需要后续 Runner 队列处理。

## 十、后续演进建议

### P1:运行队列与历史

- 增加 `flow-runs` namespace 持久化。
- 保存 `FlowRunReport` 最近 N 条。
- UI 展示最近运行、节点状态和错误。

### P2:真实 Action Registry

- 将 `AppFlowActionExecutor` 中的 `match uses` 拆为可注册 action map。
- 为 action 增加 metadata: required inputs、outputs、risk level。
- 将 `notes/build-html` 接入真实笔记构建系统。

### P3:高风险动作确认

- 对 `git/push`、网络、文件写入、删除类动作做显式确认。
- 确认逻辑放在具体 action 或 UI 交互层,不放进 Flow YAML 模型。

### P4:表达式与条件

- 支持 `if`。
- 支持 `inputs.branch || gt.workspace.active_branch` 这类安全表达式。
- 支持白名单 predicate,例如 `gt.git.has_changes(repo)`。

### P5:调度和自动消费

- `flow_emit_event` 产生 run intent 后进入队列。
- 队列按并发策略执行。
- 支持 `schedule`、`file_watch`、`store_changed` 的实际触发源。

### P6:可视化构造

- UI 使用 `flow_build_draft` 形成“选择事件 -> 选择节点 -> 预览 YAML -> 诊断 -> 保存”的闭环。
- 节点列表页展示某个 action 的 inputs、outputs 和风险提示。
- Flow 详情页增加“运行一次”和运行结果面板。
