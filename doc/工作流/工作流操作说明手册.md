# 工作流操作说明手册

> 本手册面向 GitTributary 当前已落地的工作流能力,说明如何创建、保存、运行 Flow,以及如何理解执行结果。

---

## 一、当前可用能力

当前 Flow 已经可以完成:

| 能力 | 当前状态 |
| --- | --- |
| YAML 方式创建 Flow | 已支持 |
| 保存 Flow 到本地 Store | 已支持 |
| 启用 / 暂停 Flow | 已支持 |
| 手动运行一次 | 已支持 |
| 查看最近一次运行结果 | 已支持 |
| step outputs 传递 | 已支持 |
| Git 提交 / 推送节点 | 已支持 |
| 定时触发 `schedule` | 可解析和展示,暂未自动调度 |
| 事件触发后自动执行 | 暂未接队列,事件池只生成 run intent |
| 运行历史持久化 | 暂未支持,当前 UI 只保留最近一次运行结果 |

## 二、基本概念

### 2.1 Flow

Flow 是一份 YAML 描述:

- 叫什么。
- 是否启用。
- 通过什么入口触发。
- 依次执行哪些节点。
- 每个节点需要哪些输入。

### 2.2 本地仓库和远程仓库

Flow 的 Git 节点当前操作的是本地仓库路径,不是远程 URL。

示例:

```yaml
repo: /Users/mi/rust_code/gt-test
```

这里的 `repo` 必须是本机上能被 Git 打开的目录。

远程仓库 URL 例如:

```text
https://github.com/ChenXuRiYue/gt-test.git
```

它属于本地仓库 `.git/config` 里的 remote 配置。推送节点使用的是 remote 名称,例如:

```yaml
remote: origin
```

不是远程 URL。

### 2.3 输入和输出

节点输入写在 `with` 中:

```yaml
with:
  repo: /Users/mi/rust_code/gt-test
  message: "chore: auto sync ${{ gt.now }}"
```

节点输出不需要手动传。节点执行后会返回 outputs,Runner 自动挂到:

```text
steps.<step_id>.outputs.<field>
```

下游节点这样引用:

```yaml
branch: ${{ steps.commit.outputs.branch }}
```

## 三、创建一个 Flow

### 3.1 打开入口

在工作流模块中进入“流”视图。

如果当前没有 Flow,可以点击“添加 Flow”。如果已有文件夹,也可以在文件夹上添加 Flow。

### 3.2 编辑 YAML

点击“添加 Flow”后会进入 YAML 编辑模式。

一个最小可运行例子:

```yaml
name: gt-test 手动提交推送

gt:
  id: flow.gt_test_commit_push
  enabled: true
  description: 手动提交 gt-test 仓库的全部变更并推送到 origin/main

on:
  workflow_dispatch:

jobs:
  commit_and_push:
    runs-on: gittributary-local
    steps:
      - id: commit
        uses: gittributary/git/commit-all@v1
        with:
          repo: /Users/mi/rust_code/gt-test
          message: "chore: auto sync ${{ gt.now }}"

      - id: push
        uses: gittributary/git/push@v1
        with:
          repo: /Users/mi/rust_code/gt-test
          remote: origin
          branch: ${{ steps.commit.outputs.branch }}
```

### 3.3 保存

点击“保存”。

保存时会做 YAML 解析。如果 YAML 结构错误、Flow ID 缺失、字段类型不合法,编辑器会显示错误。

当前 `flow_save` 主要做语法解析和摘要生成。节点缺输入等更完整诊断由构造草稿能力提供,后续还需要接到保存前体验中。

## 四、启用和暂停

Flow 详情顶部有启停开关。

状态含义:

| 状态 | 含义 |
| --- | --- |
| 已启用 | 允许手动运行和后续调度运行 |
| 已暂停 | Runner 会跳过执行,返回 `skipped` |

注意: 当前历史数据中可能出现外层 `record.enabled` 和 YAML 内 `gt.enabled` 不一致的旧记录。新的 Flow 建议保持一致:

```yaml
gt:
  enabled: true
```

并在 UI 中也保持“已启用”。

## 五、运行一个 Flow

### 5.1 手动运行

在 Flow 详情顶部点击“运行一次”。

当前 UI 调用:

```text
flow_run(id, request: { intent: null, inputs: {} })
```

也就是说,当前手动运行不会弹出额外参数输入面板。如果 Flow 需要动态参数,暂时需要写死在 YAML 中,或使用表达式读取工作区上下文。

### 5.2 可用表达式

当前 `with` 字符串中支持这些表达式:

| 表达式 | 说明 |
| --- | --- |
| `${{ gt.now }}` | 当前运行时间 |
| `${{ gt.workspace.active_repo }}` | Git 模块当前打开的本地仓库 |
| `${{ gt.workspace.active_branch }}` | Git 模块当前分支 |
| `${{ event.data.xxx }}` | 事件数据 |
| `${{ inputs.xxx }}` | 运行请求传入参数 |
| `${{ steps.<id>.outputs.<name> }}` | 前置节点输出 |

示例:

```yaml
message: "chore: auto sync ${{ gt.now }}"
branch: ${{ steps.commit.outputs.branch }}
```

不支持:

- 逻辑表达式。
- 函数调用。
- `if` 条件。
- shell 命令。
- 任意脚本执行。

## 六、查看执行情况

运行后详情页会出现“最近执行”区域。

它展示:

| 信息 | 说明 |
| --- | --- |
| 状态 | 整个 Flow 的状态 |
| 触发 | 当前触发来源,手动运行通常是 `workflow_dispatch` |
| Run ID | 本次运行 ID |
| 原因 | 运行原因,手动运行通常是 `manual_run` |
| Job 状态 | 每个 job 的执行状态 |
| 节点状态 | 每个 step 的执行状态 |
| message | 节点执行返回的信息 |
| error | 节点失败原因 |
| outputs | 节点输出 JSON |

状态含义:

| 状态 | 含义 |
| --- | --- |
| `succeeded` | 执行成功 |
| `failed` | 执行失败 |
| `skipped` | 被跳过 |
| `running` | 执行中 |
| `pending` | 等待中,当前 P0 运行链路较少出现 |

## 七、常见问题

### 7.1 为什么 `commit-all` 显示 `nothing_to_commit`?

这不是缺提交信息。

含义是 Git 工作区没有任何可提交变更。节点会返回:

```json
{"branch":"main","commit":null}
```

并标记为 `skipped`。

如果要生成提交,需要先在目标本地仓库中修改、新增或删除文件。

### 7.2 为什么 `repo` 不是我配置的远程仓库 URL?

因为 `repo` 需要本地仓库路径。

正确:

```yaml
repo: /Users/mi/rust_code/gt-test
```

错误:

```yaml
repo: https://github.com/ChenXuRiYue/gt-test.git
```

远程 URL 属于本地仓库的 remote 配置。推送时写 remote 名:

```yaml
remote: origin
```

### 7.3 为什么 `schedule` 没自动运行?

当前 `schedule` 可以解析和展示,但自动调度器还没有接上。

现在真正可用的入口是:

```yaml
on:
  workflow_dispatch:
```

也就是点击“运行一次”。

### 7.4 为什么下游引用输出失败?

下游只能引用前置节点的 outputs。

正确:

```yaml
- id: commit
  uses: gittributary/git/commit-all@v1
  with:
    repo: /Users/mi/rust_code/gt-test
    message: "chore: auto sync ${{ gt.now }}"

- id: push
  uses: gittributary/git/push@v1
  with:
    repo: /Users/mi/rust_code/gt-test
    remote: origin
    branch: ${{ steps.commit.outputs.branch }}
```

错误:

```yaml
- id: push
  uses: gittributary/git/push@v1
  with:
    branch: ${{ steps.commit.outputs.branch }}

- id: commit
  uses: gittributary/git/commit-all@v1
```

因为 `push` 在 `commit` 之前,无法读取后置节点输出。

### 7.5 为什么保存能成功,运行却失败?

保存阶段主要保证 YAML 能解析。运行阶段才会真正:

- 打开本地仓库。
- 检查路径是否存在。
- 复制文件。
- 提交 Git。
- 推送远程。
- 渲染表达式。

因此运行失败通常需要看“最近执行”中具体节点的 `error`。

## 八、推荐模板

### 8.1 当前打开仓库自动提交

```yaml
name: 当前仓库手动提交

gt:
  id: flow.active_repo_commit
  enabled: true
  description: 手动提交当前打开仓库的全部变更

on:
  workflow_dispatch:

jobs:
  commit:
    runs-on: gittributary-local
    steps:
      - id: commit
        uses: gittributary/git/commit-all@v1
        with:
          repo: ${{ gt.workspace.active_repo }}
          message: "chore: auto sync ${{ gt.now }}"
```

name: gt-test 手动提交推送

      - id: commit

          message: "chore: auto sync ${{ gt.now }}"

      - id: push
