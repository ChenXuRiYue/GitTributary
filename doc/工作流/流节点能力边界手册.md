# 流节点能力边界手册

> 本手册描述当前内置 Flow 节点的输入、输出、真实执行行为和边界。
> 它用于判断某个需求能否直接通过现有节点完成,以及写 YAML 时应该如何连接节点。

---

## 一、节点通用约定

### 1.1 节点是什么

Flow 中的每个 step 会被编译成一个节点实例。

示例:

```yaml
- id: commit
  uses: gittributary/git/commit-all@v1
  with:
    repo: /Users/mi/rust_code/gt-test
    message: "chore: auto sync ${{ gt.now }}"
```

其中:

| 字段 | 说明 |
| --- | --- |
| `id` | step ID,后续引用 outputs 时需要 |
| `uses` | 节点动作标识 |
| `with` | 节点输入 |

### 1.2 输入

节点输入来自 `with`。Runner 会先渲染表达式,再把最终字符串交给节点执行器。

当前输入值以字符串为主。布尔值等类型也会从字符串解析,例如:

```yaml
non_empty: "true"
```

### 1.3 输出

节点输出由 action 执行后返回。用户不需要手动传 outputs。

下游通过:

```text
${{ steps.<id>.outputs.<name> }}
```

引用前置节点输出。

示例:

```yaml
branch: ${{ steps.commit.outputs.branch }}
```

### 1.4 状态

节点运行状态:

| 状态 | 说明 |
| --- | --- |
| `succeeded` | 节点成功 |
| `failed` | 节点失败,当前 run 停止 |
| `skipped` | 节点被跳过,例如没有可提交内容 |

当前 Runner 在节点 `failed` 时停止后续执行。对 `skipped` 的传播还比较基础,后续节点仍可能继续执行。

## 二、表达式边界

当前节点输入支持这些表达式:

| 表达式 | 示例 |
| --- | --- |
| 当前时间 | `${{ gt.now }}` |
| 当前打开仓库 | `${{ gt.workspace.active_repo }}` |
| 当前分支 | `${{ gt.workspace.active_branch }}` |
| 事件数据 | `${{ event.data.repo }}` |
| 运行参数 | `${{ inputs.repo }}` |
| 前置节点输出 | `${{ steps.context.outputs.output_dir }}` |

当前不支持:

- `${{ a || b }}` 这类逻辑表达式。
- `${{ gt.git.has_changes(...) }}` 这类函数调用。
- `if` 条件执行。
- `needs` 跨 job 依赖。
- shell `run`。
- 动态矩阵。

表达式引用不存在时,节点会失败,不会静默变成空字符串。

## 四、节点明细

### 4.1 `gittributary/workspace/resolve-publish-context@v1`

用途: 解析发布上下文,产出源仓库、目标仓库、目标分支和输出目录。

输入:

| 输入 | 类型 | 说明 |
| --- | --- | --- |
| `source_repo` | string | 源仓库本地路径 |
| `target_repo` | string | 目标仓库本地路径 |
| `target_branch` | string | 目标分支 |

输出:

| 输出 | 类型 | 说明 |
| --- | --- | --- |
| `source_repo` | string | 最终源仓库路径 |
| `target_repo` | string | 最终目标仓库路径 |
| `target_branch` | string | 最终目标分支 |
| `output_dir` | string | 默认输出目录 |

行为:

- 如果输入为空,执行器会从工作区上下文补默认值。
- `source_repo` 默认取 `gt.workspace.active_repo`。
- `target_repo` 默认等于 `source_repo`。
- `target_branch` 默认取 `gt.workspace.active_branch`,没有则为 `main`。
- `output_dir` 形如 `<source_repo>/.gittributary/output`。

边界:

- 不检查仓库是否真实存在。
- 不读取远程仓库 URL。
- 不创建目录。
- 不做 Git 操作。

示例:

```yaml
- id: context
  uses: gittributary/workspace/resolve-publish-context@v1
  with:
    source_repo: ${{ gt.workspace.active_repo }}
    target_repo: ${{ gt.workspace.active_repo }}
    target_branch: ${{ gt.workspace.active_branch }}
```

### 4.2 `gittributary/notes/build-html@v1`

用途: 构建笔记 HTML。

输入:

| 输入 | 类型 | 说明 |
| --- | --- | --- |
| `repo` | string | 笔记仓库本地路径 |
| `output` | string | 输出目录 |

输出:

| 输出 | 类型 | 说明 |
| --- | --- | --- |
| `html_dir` | string | HTML 产物目录 |

行为:

- 当前是占位实现。
- 不真正构建笔记。
- 返回 `html_dir = output`。

边界:

- 未接真实笔记构建系统。
- 不校验 `repo` 是否存在。
- 不校验 `output` 是否存在或非空。
- 如果后续要确保产物存在,需要接 `files/assert-exists`。

示例:

```yaml
- id: build
  uses: gittributary/notes/build-html@v1
  with:
    repo: ${{ steps.context.outputs.source_repo }}
    output: ${{ steps.context.outputs.output_dir }}
```

### 4.3 `gittributary/files/assert-exists@v1`

用途: 校验文件或目录存在,可选校验非空。

输入:

| 输入 | 类型 | 说明 |
| --- | --- | --- |
| `path` | string | 要检查的路径 |
| `non_empty` | boolean | 是否要求非空 |

输出:

| 输出 | 类型 | 说明 |
| --- | --- | --- |
| `path` | string | 已校验路径 |

行为:

- `path` 不存在时失败。
- `non_empty` 为 `true` 时,如果路径为空则失败。
- 文件存在即视为非空。
- 目录非空才通过非空检查。

边界:

- 不创建路径。
- 不修复空目录。
- 不检查文件内容格式。

示例:

```yaml
- id: assert_build
  uses: gittributary/files/assert-exists@v1
  with:
    path: ${{ steps.build.outputs.html_dir }}
    non_empty: "true"
```

### 4.4 `gittributary/files/sync-dir@v1`

用途: 将一个目录递归复制到另一个目录。

输入:

| 输入 | 类型 | 说明 |
| --- | --- | --- |
| `from` | string | 源目录 |
| `to` | string | 目标目录 |

输出:

| 输出 | 类型 | 说明 |
| --- | --- | --- |
| `changed_count` | number | 复制文件数量 |

行为:

- `from` 必须存在。
- 自动创建 `to` 目录。
- 递归复制文件。
- 已存在目标文件会被覆盖。

边界:

- 不删除目标目录中源目录已不存在的旧文件。
- 不做增量 diff。
- 不处理冲突策略。
- 不保留复杂元数据语义。

示例:

```yaml
- id: sync
  uses: gittributary/files/sync-dir@v1
  with:
    from: ${{ steps.build.outputs.html_dir }}
    to: /Users/mi/rust_code/gt-test/public
```

### 4.5 `gittributary/git/commit-all@v1`

用途: 在指定本地仓库中暂存全部变更并提交。

输入:

| 输入 | 类型 | 说明 |
| --- | --- | --- |
| `repo` | string | 本地仓库路径 |
| `message` | string | 提交信息 |

输出:

| 输出 | 类型 | 说明 |
| --- | --- | --- |
| `commit` | string/null | 新提交 ID。无变更时为 `null` |
| `branch` | string | 当前分支 |

行为:

- 打开 `repo` 本地仓库。
- 执行 stage all。
- 使用 `message` 创建提交。
- 没有可提交变更时返回 `skipped`。
- 无变更时 message 为 `nothing_to_commit`。

边界:

- `repo` 必须是本地路径,不能是远程 URL。
- 会暂存所有变更,没有 path 过滤。
- 不支持 amend。
- 不支持选择性提交。
- 不自动 push。
- 不自动跳过后续节点。

示例:

```yaml
- id: commit
  uses: gittributary/git/commit-all@v1
  with:
    repo: /Users/mi/rust_code/gt-test
    message: "chore: auto sync ${{ gt.now }}"
```

无变更时输出示例:

```json
{"branch":"main","commit":null}
```

### 4.6 `gittributary/git/push@v1`

用途: 推送指定本地仓库的指定分支到指定 remote。

输入:

| 输入 | 类型 | 说明 |
| --- | --- | --- |
| `repo` | string | 本地仓库路径 |
| `remote` | string | remote 名称,例如 `origin` |
| `branch` | string | 分支名 |

输出:

| 输出 | 类型 | 说明 |
| --- | --- | --- |
| `remote` | string | 推送 remote |
| `branch` | string | 推送分支 |

行为:

- 打开 `repo` 本地仓库。
- 从 Store 解析认证信息。
- 调用 Git push。

边界:

- `remote` 是远程名,不是远程 URL。
- 不自动创建 remote。
- 不自动设置 upstream。
- 不自动 pull / rebase。
- 不处理冲突。
- 远端拒绝时节点失败。

示例:

```yaml
- id: push
  uses: gittributary/git/push@v1
  with:
    repo: /Users/mi/rust_code/gt-test
    remote: origin
    branch: ${{ steps.commit.outputs.branch }}
```

### 4.7 `gittributary/store/sync-now@v1`

用途: 立即同步数据中心配置。

输入: 无。

输出:

| 输出 | 类型 | 说明 |
| --- | --- | --- |
| `message` | string | 同步结果描述 |

行为:

- 复用数据中心同步链路。
- 需要已有数据中心配置。

边界:

- 不接收仓库路径输入。
- 不用于同步任意 Git 仓库。
- 失败原因依赖数据中心绑定状态、token、远程连通性等。

示例:

```yaml
- id: sync_store
  uses: gittributary/store/sync-now@v1
```

### 4.8 `gittributary/ui/notify@v1`

用途: 将 Flow 结果反馈给用户或界面。

输入:

| 输入 | 类型 | 说明 |
| --- | --- | --- |
| `title` | string | 通知标题 |
| `message` | string | 通知内容 |

输出: 无。

行为:

- 当前不会弹系统通知。
- 会把 `title: message` 写入节点运行 message。

边界:

- 不接系统通知中心。
- 不持久化通知。
- 不支持通知动作按钮。

示例:

```yaml
- id: notify
  uses: gittributary/ui/notify@v1
  with:
    title: Flow 完成
    message: "提交结果: ${{ steps.commit.outputs.commit }}"
```

## 五、组合示例

### 5.1 当前仓库提交

```yaml
name: 当前仓库手动提交

gt:
  id: flow.active_repo_commit
  enabled: true
  description: 手动提交当前打开仓库的全部变更

on:
  workflow_dispatch:

jobs:
  commit:
    runs-on: gittributary-local
    steps:
      - id: commit
        uses: gittributary/git/commit-all@v1
        with:
          repo: ${{ gt.workspace.active_repo }}
          message: "chore: auto sync ${{ gt.now }}"
```

### 5.2 构建、校验、同步、提交

```yaml
name: 笔记产物同步

gt:
  id: flow.notes_publish_local
  enabled: true
  description: 构建笔记产物后同步到目标目录并提交

on:
  workflow_dispatch:

jobs:
  publish:
    runs-on: gittributary-local
    steps:
      - id: context
        uses: gittributary/workspace/resolve-publish-context@v1
        with:
          source_repo: ${{ gt.workspace.active_repo }}
          target_repo: ${{ gt.workspace.active_repo }}
          target_branch: ${{ gt.workspace.active_branch }}

      - id: build
        uses: gittributary/notes/build-html@v1
        with:
          repo: ${{ steps.context.outputs.source_repo }}
          output: ${{ steps.context.outputs.output_dir }}

      - id: assert_build
        uses: gittributary/files/assert-exists@v1
        with:
          path: ${{ steps.build.outputs.html_dir }}
          non_empty: "true"

      - id: sync
        uses: gittributary/files/sync-dir@v1
        with:
          from: ${{ steps.build.outputs.html_dir }}
          to: ${{ steps.context.outputs.target_repo }}/public

      - id: commit
        uses: gittributary/git/commit-all@v1
        with:
          repo: ${{ steps.context.outputs.target_repo }}
          message: "docs: publish notes ${{ gt.now }}"
```

注意: 当前 `notes/build-html` 仍是占位实现,这个示例主要展示节点连接方式。
