# 流体系设计规范

> GitTributary Flow 采用 **GitHub Actions workflow syntax 的本地子集**作为流描述语言,并采用 **CloudEvents** 作为内部事件信封。
> 目标: 不自创一门新 DSL。让人、AI、工具链都能用已有工作流心智理解 GitTributary 的自动化系统。

参考:

- GitHub Actions workflow syntax: https://docs.github.com/en/actions/reference/workflows-and-actions/workflow-syntax
- CloudEvents specification: https://github.com/cloudevents/spec/blob/main/cloudevents/spec.md

---

## 一、定位

Flow 是 GitTributary 的本地自动化层。

它负责任务编排,不直接拥有 Git、复习、导出、脱敏、同步等业务逻辑。业务能力以本地 action 的方式注册,Flow 只描述:

```
CloudEvent 事件
  -> workflow 的 on 匹配
  -> job.if / step.if 条件过滤
  -> steps 顺序执行本地 actions
  -> 输出新的 CloudEvent
  -> 其他 workflow 继续响应
```

一句话:

> GitTributary Flow = GitHub Actions 子集 + GitTributary 本地 action registry + CloudEvents 事件模型。

---

## 二、为什么选 GitHub Actions 子集

| 目标 | 选择原因 |
| --- | --- |
| AI 容易生成 | 大模型普遍熟悉 `on` / `jobs` / `steps` / `uses` / `with` / `if` |
| 开发者容易理解 | 绝大多数工程师见过 GitHub Actions YAML |
| 语法足够表达 | 定时、手动、事件触发、权限、并发、条件、步骤都已有成熟心智 |
| 实现可控 | 只实现本地需要的安全子集,不接收完整 GitHub Actions 的全部复杂度 |
| 未来可迁移 | 部分 workflow 可被 AI 或工具迁移到真正的 GitHub Actions / 其他 CI DSL |

不选择完整 GitHub Actions 的原因:

- GitTributary 是桌面本地运行,不是 GitHub 托管 runner。
- 不需要 `ubuntu-latest`、matrix、services、container 等 CI 能力。
- 出于安全原因,默认不开放任意 shell `run`。
- 自定义事件、文件监听、store 变化是 GitTributary 本地语义,不是 GitHub 官方事件。

## 三、兼容边界

### 兼容目标

Flow 文件尽量长得像 GitHub Actions workflow:

```yaml
name: 自动备份笔记

on:
  schedule:
    - cron: "*/30 * * * *"
  workflow_dispatch:

permissions:
  git: [status, commit, push]
  store: [read]
  network: true

concurrency:
  group: repo-${{ gt.workspace.active_repo }}
  cancel-in-progress: false

jobs:
  backup:
    runs-on: gittributary-local
    if: ${{ gt.git.has_changes(gt.workspace.active_repo) }}
    timeout-minutes: 10
    steps:
      - id: commit
        uses: gittributary/git/commit-all@v1
        with:
          repo: ${{ gt.workspace.active_repo }}
          message: "data: auto-sync ${{ gt.now }}"

      - id: push
        uses: gittributary/git/push@v1
        with:
          repo: ${{ gt.workspace.active_repo }}
          remote: origin
          branch: ${{ steps.commit.outputs.branch }}
```

### 不承诺完全兼容

这个 YAML **不是**交给 GitHub 运行的 workflow。它是 GitTributary 本地 runner 执行的 workflow。

兼容的是语法心智和字段形状,不是 GitHub 托管平台的运行时。

### YAML 解析要求

`on` 在 YAML 1.1 中可能被解析成布尔值。实现时必须:

- 使用 YAML 1.2 语义的 parser,或
- 对顶层 `on` 做特殊兼容处理。

## 四、文件与存储

Flow 属于用户配置,归数据中心管理。

推荐两种来源:

| 来源 | 说明 |
| --- | --- |
| `~/.git-tributary/flows/*.yml` | 人可直接编辑的本地 workflow 文件 |
| `gt-store` 的 `flows` namespace | UI / AI 生成后保存的结构化副本 |

数据中心 key:

| namespace | key | value |
| --- | --- | --- |
| `flows` | `workflow.<id>` | 原始 YAML + 解析后的摘要 |
| `flow-runs` | `run.<run_id>` | 最近运行摘要 |
| `flow-events` | `event.<event_id>` | 可选事件索引 |

保留策略:

- workflow 配置可同步。
- run history 和 event log 默认只保留本地最近 1000 条。
- 敏感输入不写入 workflow,只引用 `private.credentials` 或由 action 内部读取。

## 五、顶层字段子集

支持字段:

| 字段 | 必填 | GitHub Actions 对应 | GitTributary 语义 |
| --- | --- | --- | --- |
| `name` | 是 | `name` | workflow 展示名 |
| `run-name` | 否 | `run-name` | 本次运行展示名,支持表达式 |
| `on` | 是 | `on` | 触发器 |
| `permissions` | 否 | `permissions` | 本地权限声明,字段扩展为 GitTributary 权限域 |
| `env` | 否 | `env` | workflow 级变量 |
| `concurrency` | 否 | `concurrency` | 并发锁 |
| `jobs` | 是 | `jobs` | 任务集合 |
| `gt` | 否 | 无 | GitTributary 扩展元信息 |

`gt` 扩展字段:

```yaml
gt:
  id: flow.auto_backup
  enabled: true
  description: 定时提交并推送当前笔记仓库
  tags: [git, backup]
```

规则:

- `gt.id` 是稳定 ID。缺省时由文件名生成。
- `gt.enabled` 默认 `true`。
- `gt.*` 是唯一允许的顶层扩展区。避免污染 GitHub Actions 字段心智。

## 六、on 触发器子集

### 1. schedule

与 GitHub Actions 同形:

```yaml
on:
  schedule:
    - cron: "0 9 * * *"
      timezone: Asia/Shanghai
      missed: skip
```

GitTributary 扩展:

| 字段 | 默认值 | 说明 |
| --- | --- | --- |
| `timezone` | 系统时区 | 本地定时使用的时区 |
| `missed` | `skip` | 应用休眠错过任务后如何处理:`skip` / `run-once` |

### 2. workflow_dispatch

手动触发,与 GitHub Actions 同形:

```yaml
on:
  workflow_dispatch:
    inputs:
      branch:
        description: 推送分支
        required: false
        default: main
```

用于按钮、命令面板、AI 临时执行。

### 3. GitTributary 事件触发

用 GitHub Actions 的 event map 形状承载本地事件名:

```yaml
on:
  git.commit.created:
    branches: [main]
    repositories:
      - ${{ gt.workspace.active_repo }}
```

事件名是 CloudEvents 的 `type`。

常用事件:

| 事件 | 说明 |
| --- | --- |
| `app.started` | 应用启动 |
| `git.repo.opened` | 仓库打开 |
| `git.status.changed` | 工作区状态变化 |
| `git.commit.created` | 新提交创建 |
| `git.push.completed` | 推送完成 |
| `store.key.changed` | Store key 变化 |
| `file.created` | 文件创建 |
| `file.modified` | 文件修改 |
| `file.deleted` | 文件删除 |
| `file.renamed` | 文件重命名 |
| `review.due.available` | 存在待复习内容 |
| `flow.run.succeeded` | 某个 flow 成功 |
| `flow.run.failed` | 某个 flow 失败 |

### 4. file_watch

本地扩展事件,形状仍放在 `on` 下:

```yaml
on:
  file_watch:
    paths:
      - ${{ gt.workspace.active_repo }}
    types: [created, modified, deleted, renamed]
    glob: "**/*.md"
    debounce: 5s
```

等价于监听文件系统后生成 CloudEvents:

- `file.created`
- `file.modified`
- `file.deleted`
- `file.renamed`

### 5. store_changed

本地扩展事件:

```yaml
on:
  store_changed:
    namespace: settings
    keys:
      - security.redaction.enabled
```

等价于监听 `store.key.changed`。

## 七、CloudEvents 事件模型

内部事件统一使用 CloudEvents 结构。

```json
{
  "specversion": "1.0",
  "id": "evt_01HY...",
  "source": "gittributary://gt-git",
  "type": "git.commit.created",
  "time": "2026-06-24T22:30:00+08:00",
  "subject": "repo:/Users/mi/note",
  "datacontenttype": "application/json",
  "data": {
    "repo": "/Users/mi/note",
    "branch": "main",
    "commit": "a1b2c3d"
  }
}
```

必填字段:

| 字段 | 说明 |
| --- | --- |
| `specversion` | 固定 `1.0` |
| `id` | 全局唯一事件 ID |
| `source` | 事件来源 URI |
| `type` | 事件类型,点分命名 |

建议字段:

| 字段 | 说明 |
| --- | --- |
| `time` | RFC3339 时间 |
| `subject` | 事件主体,用于审计和并发锁 |
| `datacontenttype` | 默认 `application/json` |
| `data` | 业务数据 |

Flow 运行自动发出:

- `flow.run.queued`
- `flow.run.started`
- `flow.job.started`
- `flow.job.succeeded`
- `flow.job.failed`
- `flow.step.started`
- `flow.step.succeeded`
- `flow.step.failed`
- `flow.run.succeeded`
- `flow.run.failed`
- `flow.run.skipped`

## 八、表达式子集

表达式沿用 GitHub Actions 的 `${{ ... }}` 外观,但只实现安全子集。

支持上下文:

| 上下文 | 说明 |
| --- | --- |
| `gt` | GitTributary 运行时上下文 |
| `event` | 当前 CloudEvent |
| `inputs` | `workflow_dispatch` 输入 |
| `env` | workflow / job / step 环境变量 |
| `steps` | 前序 step 输出 |
| `jobs` | 前序 job 输出 |
| `secrets` | 脱敏引用,不返回明文 |

`gt` 子字段:

| 字段 | 说明 |
| --- | --- |
| `gt.workspace.active_repo` | 当前仓库 |
| `gt.workspace.active_branch` | 当前分支 |
| `gt.profile.active` | 当前 Profile |
| `gt.now` | 当前时间 |
| `gt.git.has_changes(repo)` | 只读 predicate |
| `gt.review.has_due(repo)` | 只读 predicate |

规则:

- 表达式不能执行 shell。
- 表达式不能读任意文件。
- 表达式函数只能来自白名单 predicate。
- action 参数渲染失败时,workflow 必须失败而不是静默变空。

## 九、permissions 权限子集

保留 GitHub Actions 的 `permissions` 位置,但权限域换成 GitTributary 本地能力:

```yaml
permissions:
  git: [status, log, commit, push]
  store: [read, write]
  files: [read, write]
  network: true
  secrets: false
  ui: [notify, confirm]
```

默认权限:

```yaml
permissions:
  git: [status, log]
  store: [read]
  files: [read]
  network: false
  secrets: false
  ui: [notify]
```

规则:

- action 注册时必须声明所需权限。
- 保存 workflow 时做静态权限校验。
- 首次启用包含 `push`、`delete`、`discard`、`network`、`secrets` 的 workflow,必须让用户确认。
- `secrets: true` 只代表允许 action 内部使用凭证,表达式上下文不返回明文。

## 十、concurrency 并发子集

沿用 GitHub Actions 形状:

```yaml
concurrency:
  group: repo-${{ gt.workspace.active_repo }}
  cancel-in-progress: false
```

语义:

| 字段 | 说明 |
| --- | --- |
| `group` | 并发锁 key |
| `cancel-in-progress: true` | 新 run 替换旧 run |
| `cancel-in-progress: false` | 同组 run 排队或跳过 |

GitTributary 扩展:

```yaml
gt:
  concurrency-mode: skip
```

`gt.concurrency-mode` 可选:

- `skip`: 同组已有运行时跳过新 run。
- `queue`: 排队。
- `replace`: 取消旧 run,运行新 run。

默认 `skip`。

## 十一、jobs 子集

支持字段:

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `jobs.<job_id>` | 是 | job 定义 |
| `name` | 否 | 展示名 |
| `runs-on` | 是 | 当前仅支持 `gittributary-local` |
| `needs` | 否 | 依赖其他 job |
| `if` | 否 | job 条件 |
| `permissions` | 否 | job 级权限,不能超出顶层权限 |
| `env` | 否 | job 级变量 |
| `timeout-minutes` | 否 | job 超时 |
| `steps` | 是 | step 列表 |
| `outputs` | 否 | job 输出 |

不支持:

- `strategy.matrix`
- `container`
- `services`
- 远程 runner labels
- reusable workflow `jobs.<job_id>.uses`

并行规则:

- 没有依赖关系的 jobs 可并行执行。
- `needs` 决定 job DAG。
- P0 可先只实现单 job,但语法保留 `jobs`。

---

## 十二、steps 子集

推荐只使用 `uses`:

```yaml
steps:
  - id: commit
    name: 提交全部变更
    uses: gittributary/git/commit-all@v1
    with:
      repo: ${{ gt.workspace.active_repo }}
      message: "data: auto-sync ${{ gt.now }}"
```

支持字段:

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `id` | 否 | 后续引用输出时必须提供 |
| `name` | 否 | 展示名 |
| `uses` | 是 | 本地 action 引用 |
| `with` | 否 | action 参数 |
| `if` | 否 | step 条件 |
| `env` | 否 | step 级变量 |
| `continue-on-error` | 否 | 失败后是否继续 |
| `timeout-minutes` | 否 | step 超时 |

### run 的安全策略

GitHub Actions 支持 `run`,但 GitTributary 默认不支持任意 shell:

```yaml
steps:
  - run: echo "disabled by default"
```

如后续要支持,必须满足:

- workflow 显式声明 `permissions.shell: true`。
- 用户首次启用时确认。
- UI 明确展示命令内容。
- run 环境被限制在当前仓库或临时目录。

P0/P1 阶段不实现 `run`。

## 十三、本地 action 引用规范

`uses` 使用接近 GitHub Actions 的格式:

```
gittributary/<domain>/<action>@v<major>
```

示例:

| uses | 说明 |
| --- | --- |
| `gittributary/git/has-changes@v1` | 判断仓库是否有变更 |
| `gittributary/git/commit-all@v1` | 暂存全部并提交 |
| `gittributary/git/push@v1` | 推送分支 |
| `gittributary/store/sync-now@v1` | 同步配置仓库 |
| `gittributary/review/due@v1` | 判断是否有待复习 |
| `gittributary/review/export-pdf@v1` | 导出复习 PDF |
| `gittributary/security/scan@v1` | 安全扫描 |
| `gittributary/security/redact@v1` | 脱敏 |
| `gittributary/ui/notify@v1` | 通知 |
| `gittributary/ui/confirm@v1` | 用户确认 |

Action metadata 参考 GitHub action metadata 的心智,但运行时是本地 Rust/TS 函数:

```yaml
name: Git Push
description: Push a branch to remote
inputs:
  repo:
    required: true
    type: string
  remote:
    required: true
    type: string
  branch:
    required: true
    type: string
outputs:
  remote:
    type: string
  branch:
    type: string
runs:
  using: gittributary
  action: git.push
permissions:
  git: [push]
  network: true
```

## 十四、AI 生成规范

AI 生成 Flow 时必须生成 GitHub Actions 子集 YAML,不要生成旧版自定义 DSL。

输出格式:

```json
{
  "status": "ready",
  "workflow": "name: ...",
  "questions": []
}
```

或:

```json
{
  "status": "need_input",
  "workflow": "name: ...",
  "questions": [
    {
      "field": "jobs.backup.steps.push.with.remote",
      "question": "要推送到哪个远程? 常见值是 origin。"
    }
  ]
}
```

AI 规则:

1. 使用 `name` / `on` / `permissions` / `concurrency` / `jobs` / `steps`。
2. 动作只使用 `gittributary/*@v1` 已注册 action。
3. 不生成 `run`,除非用户明确要求 shell 且系统允许。
4. 高权限 workflow 默认 `gt.enabled: false`。
5. 不确定的路径、远程、分支使用 `${{ gt.* }}` 或提出问题,不要编造。
6. 生成后必须调用 `flow_validate`。

自然语言:

> 每天下午 6 点,如果笔记有变更,自动提交并推送。

AI 应生成:

```yaml
name: 每日晚间备份

gt:
  id: flow.daily_evening_backup
  enabled: false
  description: 每天 18:00 检查当前仓库,有变更则提交并推送

on:
  schedule:
    - cron: "0 18 * * *"
      timezone: Asia/Shanghai
  workflow_dispatch:

permissions:
  git: [status, commit, push]
  store: [read]
  network: true

concurrency:
  group: repo-${{ gt.workspace.active_repo }}
  cancel-in-progress: false

jobs:
  backup:
    runs-on: gittributary-local
    if: ${{ gt.git.has_changes(gt.workspace.active_repo) }}
    timeout-minutes: 10
    steps:
      - id: commit
        uses: gittributary/git/commit-all@v1
        with:
          repo: ${{ gt.workspace.active_repo }}
          message: "data: auto-sync ${{ gt.now }}"

      - id: push
        uses: gittributary/git/push@v1
        with:
          repo: ${{ gt.workspace.active_repo }}
          remote: origin
          branch: ${{ steps.commit.outputs.branch }}
```

## 十五、典型流示例

### 1. 今日复习提醒

```yaml
name: 每日复习提醒

gt:
  id: flow.review_daily
  enabled: true

on:
  schedule:
    - cron: "0 9 * * *"
      timezone: Asia/Shanghai

permissions:
  git: [log]
  store: [read]
  ui: [notify]

jobs:
  review:
    runs-on: gittributary-local
    if: ${{ gt.review.has_due(gt.workspace.active_repo) }}
    steps:
      - id: notify
        uses: gittributary/ui/notify@v1
        with:
          title: 今日待复习
          message: 有新的笔记需要复习。
```

### 2. 每周导出复习 PDF

```yaml
name: 每周导出复习 PDF

gt:
  id: flow.export_weekly_review
  enabled: true

on:
  schedule:
    - cron: "0 20 * * 0"
      timezone: Asia/Shanghai
  workflow_dispatch:

permissions:
  git: [log]
  store: [read]
  files: [read, write]
  ui: [notify]

jobs:
  export:
    runs-on: gittributary-local
    timeout-minutes: 20
    steps:
      - id: pdf
        uses: gittributary/review/export-pdf@v1
        with:
          repo: ${{ gt.workspace.active_repo }}
          range: this_week
          output: ${{ gt.workspace.active_repo }}/exports/review-${{ gt.now }}.pdf

      - id: notify
        uses: gittributary/ui/notify@v1
        with:
          title: 复习 PDF 已导出
          message: ${{ steps.pdf.outputs.output }}
```

### 3. commit 后自动推送远程博客

```yaml
name: 提交后发布博客

gt:
  id: flow.publish_blog_after_commit
  enabled: false

on:
  git.commit.created:
    branches: [main]
    repositories:
      - ${{ gt.workspace.active_repo }}

permissions:
  git: [log, push]
  store: [read]
  network: true
  ui: [notify]

concurrency:
  group: publish-${{ gt.workspace.active_repo }}
  cancel-in-progress: false

jobs:
  publish:
    runs-on: gittributary-local
    timeout-minutes: 30
    steps:
      - id: push_pages
        uses: gittributary/git/push@v1
        with:
          repo: ${{ gt.workspace.active_repo }}
          remote: origin
          branch: gh-pages

      - id: notify
        uses: gittributary/ui/notify@v1
        with:
          title: 博客已推送
          message: gh-pages 已推送到 origin。
```

后续有静态站点构建能力后,在 `push_pages` 前插入:

```yaml
      - id: build
        uses: gittributary/site/build@v1
```

### 4. 定时同步配置

```yaml
name: 定时同步配置

gt:
  id: flow.store_periodic_sync
  enabled: true

on:
  schedule:
    - cron: "*/15 * * * *"

permissions:
  store: [read, write]
  git: [status, commit, push]
  network: true

concurrency:
  group: store-sync
  cancel-in-progress: false

jobs:
  sync:
    runs-on: gittributary-local
    timeout-minutes: 10
    steps:
      - id: sync
        uses: gittributary/store/sync-now@v1
```

### 5. 监听文件增删改并自动提交

```yaml
name: 监听笔记自动提交

gt:
  id: flow.watch_notes_auto_commit
  enabled: false
  concurrency-mode: queue

on:
  file_watch:
    paths:
      - ${{ gt.workspace.active_repo }}
    types: [created, modified, deleted, renamed]
    glob: "**/*.md"
    debounce: 5s

permissions:
  git: [status, commit]
  store: [read]
  files: [read]

concurrency:
  group: repo-${{ gt.workspace.active_repo }}
  cancel-in-progress: false

jobs:
  commit:
    runs-on: gittributary-local
    if: ${{ gt.git.has_changes(gt.workspace.active_repo) }}
    timeout-minutes: 10
    steps:
      - id: commit
        uses: gittributary/git/commit-all@v1
        with:
          repo: ${{ gt.workspace.active_repo }}
          message: "notes: auto-save ${{ gt.now }}"
```

### 6. 手动安全推送

```yaml
name: 安全推送

gt:
  id: flow.safe_push
  enabled: true

on:
  workflow_dispatch:
    inputs:
      branch:
        description: 推送分支
        required: false

permissions:
  git: [status, push]
  files: [read, write]
  store: [read]
  network: true
  ui: [confirm, notify]

concurrency:
  group: repo-${{ gt.workspace.active_repo }}
  cancel-in-progress: false

jobs:
  push:
    runs-on: gittributary-local
    timeout-minutes: 15
    steps:
      - id: scan
        uses: gittributary/security/scan@v1
        with:
          repo: ${{ gt.workspace.active_repo }}
          target: staged_diff

      - id: redact
        if: ${{ steps.scan.outputs.risk == 'high' }}
        uses: gittributary/security/redact@v1
        with:
          repo: ${{ gt.workspace.active_repo }}
          findings: ${{ steps.scan.outputs.findings }}

      - id: confirm
        if: ${{ steps.scan.outputs.risk != 'none' }}
        uses: gittributary/ui/confirm@v1
        with:
          title: 推送前确认
          message: ${{ steps.scan.outputs.summary }}

      - id: push
        uses: gittributary/git/push@v1
        with:
          repo: ${{ gt.workspace.active_repo }}
          remote: origin
          branch: ${{ inputs.branch || gt.workspace.active_branch }}
```

## 十六、实现路线

| 阶段 | 范围 | 目标 |
| --- | --- | --- |
| P0 | YAML parser、WorkflowSpec 类型、validate、store 保存、手动运行单 job | 能执行一个 `workflow_dispatch` workflow |
| P1 | ActionRegistry + git/store/ui 基础 action | 自动 commit / push / sync 可用 |
| P2 | CloudEvents EventBus + flow run events | workflow 能由事件触发 |
| P3 | schedule + file_watch + store_changed | 时间、文件、配置变化触发可用 |

P0 最小闭环:

3. `flow_validate(workflow)`。

5. `flow_run(id, inputs)` 执行单 job 顺序 steps。
6. 内置 `ui.notify` 与 `store.sync-now` 两个 action。

## 十七、迁移说明

旧版自定义 FlowSpec:

```yaml
version: 1
id: flow.auto_backup
on:
  type: schedule
steps:
  - use: git.commit_all
```

迁移为 GitHub Actions 子集:

```yaml
name: 自动备份
gt:
  id: flow.auto_backup
on:
  schedule:
    - cron: "*/30 * * * *"
jobs:
  backup:
    runs-on: gittributary-local
    steps:
      - uses: gittributary/git/commit-all@v1
```

迁移原则:

- `id` -> `gt.id`
- `enabled` -> `gt.enabled`
- `on.type: schedule` -> `on.schedule`
- `on.type: manual` -> `on.workflow_dispatch`
- `on.type: event` -> `on.<event_type>`
- `policy.concurrency` -> `concurrency` + `gt.concurrency-mode`
- `steps[*].use` -> `steps[*].uses`
- `steps[*].with` 保持不变
- `when` -> `jobs.<job_id>.if` 或 `steps[*].if`
