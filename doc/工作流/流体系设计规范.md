# NoteAura Flow 规范

## 定位

Flow 是 Core 提供的本地编排能力。`na-flow` 只负责 YAML 解析、事件匹配、节点编译、
表达式解析和顺序执行，不持有 Git、文件、Store 或插件业务节点。

## 节点来源

```text
Core capability providers ----\
                               -> source inspection -> candidate registry -> atomic swap
Active plugin manifests ------/
```

- Core 能力模块声明自己真实可执行的节点。
- 活跃插件通过 `contributes.flowNodes` 声明节点和 backend method。
- 启动、安装、更新、卸载后重新巡检全部来源。
- 节点归属为 `Core` 或 `Plugin(pluginId)`，不同来源不能覆盖相同 `uses`。
- 不注册计划、占位、未实现动作。

## Workflow 子集

```yaml
name: 示例

gn:
  id: flow.example
  enabled: false

on:
  workflow_dispatch:

jobs:
  main:
    runs-on: noteaura-local
    steps:
      - id: check
        uses: noteaura/files/assert-exists@v1
        with:
          path: ${{ gn.workspace.active_repo }}
```

当前支持 `gn`、`on`、`jobs`、`steps`、`uses`、`with`，并支持引用工作区、运行输入和
前序步骤输出。未知事件、未知节点和非法前向引用由构造器诊断。

## 执行

- Core 节点由宿主执行器调用对应 Core 能力。
- 插件节点固定运行开始时的插件 generation、版本、路径、method 和权限快照，再通过
  sidecar 执行。
- 同一时间只运行一个 Flow，避免 Git、Store 和文件写入相互干扰。
- 插件在运行中更新或卸载时，旧快照执行失败，不切换到新代码。
- Flow 完成后发布成功、失败或跳过事件。

## 当前范围

当前只有手动运行链路完整可用。事件匹配已经实现；定时器、文件监听和匹配后的自动调度
仍属于后续工作。
